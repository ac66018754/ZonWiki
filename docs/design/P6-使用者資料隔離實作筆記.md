# P6 使用者資料隔離實作筆記

完成日期：2026-06-11

## 概述

實作了任務 P6（使用者資料隔離），透過 EF Core 全域查詢過濾 (Query Filters) 強制所有 IUserOwned 實體的隔離，並編寫整合測試驗證隔離。

## 實作清單

### 1. ICurrentUser 抽象介面

**檔案**：`src/ZonWiki.Domain/Common/ICurrentUser.cs`

```csharp
public interface ICurrentUser
{
    Guid UserId { get; }           // 目前使用者識別碼（未登入時為 Guid.Empty）
    string? Email { get; }         // 目前使用者電子郵件（未登入時為 null）
    bool IsAuthenticated { get; }  // 是否已驗證
}
```

- 定義在 Domain 層以供整個系統存取
- 由 HttpContext 提供身分資訊

### 2. CurrentUserService 實作

**檔案**：`src/ZonWiki.Api/Auth/CurrentUserService.cs`

- 由 API 層實作（需要 HttpContext）
- Scoped 生命週期，與 HTTP 請求同步
- 從 HttpContext 的宣告 (Claims) 擷取：
  - `UserIdClaimType` → UserId
  - `ClaimTypes.Email` → Email
  - User.Identity.IsAuthenticated → IsAuthenticated

**註冊**：在 `Program.cs` 中註冊為 Scoped 服務

```csharp
builder.Services.AddScoped<ICurrentUser, CurrentUserService>();
```

### 3. 全域查詢過濾 (Query Filters)

**檔案**：`src/ZonWiki.Infrastructure/Persistence/QueryFilterExtensions.cs`

擴充方法 `ApplyUserIsolationFilters()` 動態套用過濾：

```csharp
modelBuilder.ApplyUserIsolationFilters(currentUser);
```

**過濾邏輯**：
- 對所有實作 `IUserOwned` 的實體套用
- 條件：`UserId == 目前使用者 AND ValidFlag == true`
- 使用 LINQ 運算式樹動態建構 Lambda
- 透過反射呼叫 EF Core 的 `HasQueryFilter<T>()`

**影響實體**（自動套用隔離）：
- Note, Category, Tag, NoteCategory, NoteTag, NoteLink, NoteRevision
- Comment, TaskCard, TaskGroup, TaskRelation, NoteTaskLink
- Whiteboard, WhiteboardItem, QuickLink, CaptureItem
- Canvas, Node, Edge, InlineLink, Highlight, NodeImage, NodeRevision
- SystemPrompt, CanvasCat, CanvasCategory, CategorySystemPrompt, CanvasSystemPrompt
- AiSession, AiMessage, AiModel

### 4. ZonWikiDbContext 更新

**檔案**：`src/ZonWiki.Infrastructure/Persistence/ZonWikiDbContext.cs`

```csharp
public sealed class ZonWikiDbContext(
    DbContextOptions<ZonWikiDbContext> options,
    ICurrentUser? currentUser = null) : DbContext(options)
```

**改變**：
- 建構式接受可選的 `ICurrentUser` 參數（DI 容器會自動注入）
- `OnModelCreating()` 中調用 `ApplyUserIsolationFilters()` 若 currentUser 非 null
- 未登入情境（import/migration）時可傳 null，過濾將不套用

**使用者隔離行為**：
- 正常 HTTP 請求：DI 容器提供登入的 ICurrentUser，過濾自動套用
- Admin/Import 操作：傳 null，無過濾（需要透過其他權限機制保護）
- 忽略過濾：使用 EF Core 內建 `db.Note.IgnoreQueryFilters().ToListAsync()`

### 5. 依賴注入設定

**檔案**：`src/ZonWiki.Infrastructure/DependencyInjection.cs`

```csharp
services.AddDbContext<ZonWikiDbContext>(

```

- DbContext 為 Scoped（與 HTTP 請求同生命週期）

## 架構設計決策

### 為何使用全域查詢過濾而非應用層過濾？

1. **安全性**：無法繞過過濾（不用逐個端點檢查）
2. **一致性**：所有查詢自動套用，無遺漏
3. **效能**：DB 層過濾減少傳輸資料量
4. **可維護性**：集中管理，易於審計與修改

### 為何 ICurrentUser 在 Domain 層，實作在 API 層？

- **介面 (ICurrentUser)**：無依賴，可被任何層使用
- **實作 (CurrentUserService)**：需要 HttpContext，只能在 API 層
- **依賴反轉**：基礎設施層 (Infrastructure) 定義規約，應用層 (API) 實現

### 未登入時如何處理？

- `ICurrentUser.UserId` 傳回 `Guid.Empty`
- 過濾條件變成 `UserId == Guid.Empty AND ValidFlag == true`
- 由於正常資料的 UserId 不會是 Guid.Empty，未登入者無法看到任何資料
- 若未登入端點需存取全域資料，應調用 `.IgnoreQueryFilters()`（需權限檢查）

3. 新增審計日誌

## 已知限制與後續改進

### 當前限制

1. **匯入/匯出端點未受保護**：不需驗證即可操作任意使用者資料
   - 改進：添加管理員權限檢查
   
2. **Comment 實體處理**：Comment 有 UserId 但主要用於記錄發表者，非所有權
   - 改進：可考慮讓評論留言者 (CreatedUser) 只見自己發表的留言，但其他使用者可見該筆記的所有留言

3. **多對多表（NoteCategory, NoteTag 等）**：雖然套用了隔離，但實際上多對多的另一側可能可見
   - 改進：未來若需更細緻的權限，可在查詢層額外檢查

### 後續改進（列為待辦）

1. 為匯入/匯出端點添加管理員驗證
2. 補全 Comment 隔離邏輯（評論的可見性策略定案）
3. 添加審計日誌（誰、何時、操作了誰的資料）
4. Workspace 多租戶若未來啟用，過濾邏輯需相應擴充
