# 提問佇列（Ask Queue）— 設計決策與測試計畫

> 狀態：草案（待 sub-agent 審查測試計畫後進入 TDD）
> 日期：2026-06-25　分支：`feature/ux-batch-improvements`
> 對應需求：使用者在筆記中「框選提問」後，需要一個地方看到**過往提問佇列**（哪些已有答案、哪些等待中/失敗），並可**點擊 Queue item 跳到該筆記的框選處**。

---

## 1. 背景與關鍵發現

- 筆記「框選提問」端點 `POST /api/notes/{id}/ask-selection`（`NoteWriteEndpoints.AskSelectionHandler`）目前是**同步**呼叫 `INoteAiService.AskAboutAsync`，成功才建「答案筆記 + `NoteMark`（link）錨點」；失敗只記 log、回 500，**不留任何紀錄**。
- `AiSession`（欄位 `Status: Running/Completed/Failed`、`Kind: node/floatingnote`）目前**只給畫布節點提問用**，筆記框選提問**完全不寫 AiSession**。
- ⇒ 現況沒有任何「待處理／已答／失敗」的真實來源，這正是佇列要補的洞。這也解釋了使用者回報「框選提問後等很久沒看到答案」：同步 AI 慢或失敗，且系統無從追蹤。

## 2. 決策（已與使用者確認）

採 **方向 1：AiSession 持久化**（另兩案：真正非同步背景佇列＝工程量最大、prod 風險高；唯讀從 NoteMark 組＝無法顯示等待/失敗）。

理由：重用既有 `AiSession`（已有 Status/Kind/稽核欄位/使用者隔離），同步 AI 呼叫不變，能完整呈現「等待中/已答/失敗」，工程量適中，migration 為**純新增可空欄位**（最安全）。

「等待處理中」語意：同步呼叫進行中、或伺服器中途重啟留下的孤兒 `Running`；配合 `Failed` 即可滿足使用者「看得到哪些卡住/失敗」的需求。真正的非同步背景處理列為未來增強。

## 3. 資料模型變更（`AiSession` 新增可空欄位）

全部 nullable、節點提問留 null；以**鬆散 id 參照**保存（不設 EF 外鍵導覽，避免影響筆記刪除的 cascade/restrict 行為，查詢時自行 join）：

| 新欄位 | 型別 | 用途 |
|---|---|---|
| `NoteId` | `Guid?` | 框選提問的**來源筆記** id |
| `AnswerNoteId` | `Guid?` | 產生的**答案筆記** id（completed 才有） |
| `MarkId` | `Guid?` | 來源筆記上的 `NoteMark`（錨點）id → 點擊跳轉用 |
| `QuestionText` | `string?` (maxlen 2000) | 使用者問題（佇列顯示，免解析 PromptText） |
| `AnchorText` | `string?` (maxlen 2000) | 框選文字片段（佇列顯示） |
| `ErrorText` | `string?` | 失敗訊息（佇列顯示；不含堆疊/敏感資訊） |

- 欄位命名沿用專案既有 EF 慣例（migration 產生時複驗實際欄名）。
- Migration：`AddAskQueueFieldsToAiSession`，僅 `AddColumn`（可空），對既有資料零破壞、可回滾。

## 4. 後端設計

### 4.1 服務 `AskQueueService`（`src/ZonWiki.Api/Services/`）

- **純函式生命週期建構子（可單元測試、handler 呼叫，保持 handler 精簡）**
  - `BuildRunningNoteSession(userId, noteId, question, anchorText, promptText) : AiSession`（Kind=`floatingnote`、Status=`Running`、填稽核欄位）
  - `ApplyCompleted(session, answerNoteId, markId) : void`（Status=`Completed`、填 AnswerNoteId/MarkId、更新 UpdatedDateTime/User）
  - `ApplyFailed(session, error) : void`（Status=`Failed`、填 ErrorText）
- **讀取**：`GetQueueAsync(userId, AskQueueFilter? filter, ct) : IReadOnlyList<AskQueueItemDto>`
  - 取該使用者 `AiSession`（`ValidFlag`），依 `CreatedDateTime` 由新到舊。
  - join 來源筆記（NoteId）、答案筆記（AnswerNoteId）補 slug/title。
  - 可選 `status` 過濾、`limit`（預設 50、上限 200，避免無界查詢）。
  - 使用者隔離靠全域查詢過濾器 + 明確 `UserId == userId`。

### 4.2 端點 `GET /api/ask-queue`

- 回 `ApiResponse<IReadOnlyList<AskQueueItemDto>>`，`.RequireAuthorization()`。
- query：`?status=Running|Completed|Failed`、`?limit=`、`?kind=floatingnote|node`。

`AskQueueItemDto`：`SessionId, Status, Kind, QuestionText, AnchorText, NoteId, NoteSlug, NoteTitle, AnswerNoteId, AnswerNoteSlug, MarkId, CanvasId, AskNodeId, CreatedDateTime`。

### 4.3 改 `AskSelectionHandler`（最小侵入）

- AI 呼叫**前**：`db.AiSession.Add(BuildRunningNoteSession(...))` 先存（即使後續失敗也留痕）。
- 成功建答案筆記 + NoteMark 後：`ApplyCompleted(session, answerNote.Id, mark.Id)` 存檔。
- `catch`：`ApplyFailed(session, ex.Message 的安全摘要)` 盡力存檔，再回 500（行為不變）。

## 6. 測試計畫（**本節為 sub-agent 審查重點**）

測試風格沿用現有 `NoteWriteEndpointsTests`：xUnit + FluentAssertions + InMemory `ZonWikiDbContext`（專案尚無 WebApplicationFactory 整合測試底座）。AI 以假實作（stub `INoteAiService`）注入。

### 6.1 生命週期純函式（`AskQueueServiceLifecycleTests`）
1. `BuildRunningNoteSession` → Status=`Running`、Kind=`floatingnote`、UserId/NoteId/QuestionText/AnchorText 正確、稽核欄位已填、AnswerNoteId/MarkId 為 null。
2. `ApplyCompleted` → Status 由 Running→`Completed`、AnswerNoteId/MarkId 寫入、UpdatedDateTime 變動。
3. `ApplyFailed` → Status→`Failed`、ErrorText 寫入、**不**塞入堆疊/路徑等敏感字串（斷言不含 `at ` / 磁碟路徑樣式）。

### 6.2 讀取查詢（`AskQueueServiceReadTests`，InMemory DbContext 播種）
4. 空資料 → 回空清單。
5. 多筆 → 依 `CreatedDateTime` **由新到舊**。
6. **使用者隔離**：他人 session 不得出現。
7. 狀態映射：Running/Completed/Failed 各自正確帶出。
8. join：Completed 的 floatingnote 帶出 NoteSlug/NoteTitle/AnswerNoteSlug；來源筆記不存在時欄位為 null 不丟例外。
9. `kind=node` 的 session 帶出 CanvasId/AskNodeId（供畫布導航）。
10. `status=` 過濾只回該狀態；`limit` 生效且超過上限被夾住。
11. 已軟刪除（`ValidFlag=false`）的 session 不出現。

### 6.3 ask-selection 持久化整合（以服務層 + InMemory，不走 HTTP）
12. AI 成功 → DB 多一筆 `AiSession`：Status=`Completed`、Kind=`floatingnote`、NoteId=來源、AnswerNoteId/MarkId 指向實際建立的答案筆記與 mark。
13. AI 丟例外 → 仍留一筆 `Status=Failed`、ErrorText 有值，且**答案筆記/NoteMark 不應殘留**（或明確定義：失敗不建答案筆記）。
14. 跨使用者：A 的提問不會被 B 的佇列查到（與 #6 呼應，端到端再validate一次）。

## 8.5 測試計畫審查結果（sub-agent 對抗式審查 → 已修正）

審查找出 3 個真問題，均已驗證屬實並修正計畫：

1. **【CRITICAL】InMemory 全域過濾假陽性**：`ZonWikiDbContext` 只在「有 current user / 背景覆寫」時才套全域過濾（`ZonWikiDbContext.cs:116`），故無 user 的 InMemory 測不到全域隔離。
   → **修正**：`GetQueueAsync` 帶**明確 `UserId == userId`** 過濾（端點本來就從 auth claim 取 userId），InMemory 測此明確過濾即有效；全域過濾另由現成 Testcontainers `UserDataIsolationTests` 覆蓋（會補一筆 AiSession 案例）。
2. **【HIGH】handler 私有靜態不可測**：要 TDD #12/#13，須把 ask-selection 全流程抽進 `AskQueueService.ExecuteAskSelectionAsync(...)`，handler 改薄殼。→ 採納。
3. **【HIGH】鬆散 id 參照 + join 安全**：`GetQueueAsync` join 時須加 `note.ValidFlag` / `mark.ValidFlag`；來源/答案筆記被刪 → 對應 slug/title 回 null（前端優雅降級）。join 僅在該使用者自己的 Note 範圍內 → 不會洩漏他人標題。→ 採納並補測。

補充邊界測試（採納審查 B）：孤兒 `Running` 仍顯示等待中、答案筆記軟刪→null、mark 軟刪被濾、`QuestionText/AnchorText` 截斷至 2000、同 `CreatedDateTime` 以 `Id` 二級排序穩定、`limit` 夾住邊界、無效 `status` 值忽略、`ValidFlag=false` session 不出現、`ErrorText` 只存 `ex.Message`（不含堆疊/路徑）。

## 9. 實作演進（與上文設計的差異，2026-06-25 實作時調整）

設計定稿後，依使用者實測回饋做了以下調整，最終實作以本節為準：

1. **UI 改為 Header 下拉**（非浮動面板）：原 `AskQueuePanel`(FloatingPanel) 廢棄、刪檔；改為 `AiProcessingMenu`，放在 Header「筆記」與「搜尋框」之間，名稱「**AI處理中(n)**」（n＝進行中數），點開**就地展開清單、不跳頁**（仿全域搜尋預覽），樣式比照 `.nav-item`／`.nav-hint`（不特立獨行）。
2. **涵蓋範圍擴大到「所有 AI 生成動作」**：不只 `ask-selection`（建答案筆記）。新增通用包裝器 **`AskQueueService.TrackAiAsync<T>`**，把**便利貼提問 `ask-selection-answer`、美化 `beautify`、排版 `reformat`** 都追蹤成 AiSession（Running→Completed/Failed）。開問啦節點提問本來就有 AiSession。Kind 增加 `beautify`／`reformat`。
3. **(n) 即時更新（事件驅動）**：新增 `frontend/src/lib/aiQueue.ts` 的 `AI_QUEUE_CHANGED_EVENT` 與 `withAiQueueNotify`，包住 5 個觸發 AI 的 api 函式 → 觸發/完成即派發事件，Header menu 立即重抓；輪詢由 15s 降到 6s 當保險。
4. **孤兒／逾時保護**（同步 AI 工作卡住或被重啟打斷會永遠卡 Running）：
   - 開機清理（`Program.cs`）：啟動時把殘留 `Running` 一律 `ExecuteUpdate` 成 `Failed`。
   - 讀取時（`GetQueueAsync`）：`Running` 但建立超過 `StaleRunningMinutes`(5) → 顯示為失敗/逾時、不計入進行中。
5. **AI Provider 釐清（重要，曾誤判）**：便利貼／美化／排版的 AI = `INoteAiService`(GeminiNoteAiService) → `ResolveAsync(modelKey:null)` → **全站共用預設模型 `banana-gemini-lite`（OpenAiCompatible，Gemini 經 banana relay）**，**不是** claude CLI。claude CLI(`ClaudeCliProvider`) 只在「開問啦節點明確選 `claude-*` 模型」時才用到（prod 更會開機停用 ClaudeCli）。診斷孤兒時一度誤判成 claude CLI，更正詳見全域 CLAUDE.md 鐵則 #26。
