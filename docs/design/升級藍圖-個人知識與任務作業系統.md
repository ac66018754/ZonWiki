## 0. 一句話總結

現在的 ZonWiki 是「**Markdown 檔案為真相、DB 為唯讀鏡像、網頁唯讀**」。
要變成你要的個人知識/任務系統，核心動作是**反轉真相來源**（DB 變成可編輯的真相、Markdown 降為匯入種子 + 匯出備份），然後在既有的「浮動面板工作區」上長出 **筆記 / 任務 / 日記 / 捕捉 / 思考畫布** 五大模組——其中「思考畫布」可直接吸收你已經寫好的「開問啦」。

### 1.2 資料流（目前是單向、唯讀）

```
筆記區/*.md  ──(git push)──►  GitHub
     │
     │  .NET BackgroundService 每 5 分鐘掃描 + POST /api/sync/trigger 手動觸發
     ▼
NotesSyncService（單向 upsert：FilePath 當鍵；缺檔 → 軟刪除 ValidFlag=false）
     ▼
PostgreSQL  ──►  .NET Web API（只有 GET）  ──►  Next.js（唯讀瀏覽）
```

- **方向只有 md → DB**，`NotesSyncService.cs` 全程**沒有任何寫回 .md 的動作**（無 `File.Write*`）。
- 文章 API 只有兩個 GET（`ArticleEndpoints.cs:12,37`），**沒有 POST/PUT/DELETE 改文章內容**。
- 可寫的只有「留言」與「使用者佈建」（`CommentEndpoints.cs`），這跟「文章內容唯讀」並不衝突。

### 1.3 目前 DB Schema（4 張表）

| 表 | 重點欄位 | 用途 |
|---|---|---|
| `Category` | `ParentId`, `Name`, `FolderPath` | 分類＝資料夾（階層） |
| `Article` | `CategoryId`, `Title`, `Slug`, `FilePath`, `ContentHash`, `ContentRaw`, `ContentHtml`, `PublishedFlag` | 文章＝一個 .md 檔 |
| `Comment` | `ArticleId`, `UserId`, `Content`, **`AnchorType`(預設 "full")**, **`AnchorData`(nullable)** | 留言（已預留「範圍留言」欄位） |
| `User` | `GoogleSub`, `Email`, `DisplayName`, `AvatarUrl` | Google 登入者 |

- 全部繼承 `AuditableEntity`（`Id / CreatedDateTime / CreatedUser / UpdatedDateTime / UpdatedUser / ValidFlag` 六審計欄位）。
- `ModelBuilderExtensions.cs` 用反射自動把欄位改名成 `{Table}_{Field}`、PK/FK/Index 也照規範——**完全符合你 CLAUDE.md 的命名規則**。
- 軟刪除（`ValidFlag`）。

- **尚未實作**：網頁內編輯、wiki-link 實際跳轉解析（`[[X]]` 在同步時已轉成 `/a/slug` 連結，但前端沒有 backlink/反向連結）、全文搜尋、標籤、任務。

### 1.5 「那個 AI 到底做了什麼？」——釐清你的認知落差

你的認知是：「筆記區是舊方式，AI 寫了網頁版，之後筆記區要淘汰、資料灌進 DB」。
**實際上 AI 當初是「刻意決定不淘汰筆記區」**：它採用「混合路線 C」，明文決策「**MVP 網頁唯讀，避免雙寫衝突**」（`MVP開發進度.md:47`），把筆記區留為唯一真相來源、DB 只當索引/快取/留言儲存。

所以：

- ✅ AI 完成的：md → DB 的單向同步 + 唯讀瀏覽網頁 + Google 登入 + 留言。
- ❌ AI **沒有**做（而且是刻意不做）：把 DB 變成可編輯的真相來源、淘汰筆記區。

**你接下來要的升級，本質上就是要翻轉這個當初的架構決策。** 這不是 AI 做錯，而是 MVP 階段的合理取捨；現在需求變大了，該往下一步走。

## 2. 核心架構決策：反轉真相來源（**the pivot**）

### 2.1 為什麼非做不可

你想要的所有新功能——**網頁編輯、Todo、日記、靈感捕捉、範圍留言、AI 寫入**——全部都需要一個「**可寫**的真相儲存」。而現在「檔案為真相」之所以要把網頁鎖成唯讀，正是因為**雙寫衝突**（網頁改 DB、你又在 VS Code 改檔案，兩邊打架）。
→ 結論：要往前走，就得讓 **DB 成為真相來源**，否則永遠卡在唯讀。

### 2.2 提案：DB 為真相，Markdown 降級為「匯入種子 + 匯出備份」

| 角色 | 現在 | 升級後（提案） |
|---|---|---|
| 真相來源 | `筆記區/*.md` | **PostgreSQL** |
| 寫作介面 | VS Code + PowerShell 腳本 | **網頁（可編輯）** + 仍可選擇性匯入 |
| Markdown 檔 | 權威 | **匯入種子**（一次性）＋ **匯出備份**（DB→md 定期匯出推 GitHub，當作可攜備份/離線閱讀） |

### 2.3 過渡策略（直接回應你在《自言自語》的疑問）

你在《自言自語》問：「過渡階段，能不能照常用 VS Code 寫進 Repo，**同時也寫進資料庫**？」

**誠實的建議：不要做「兩邊都是權威」的雙寫**（這正是衝突來源，會讓你日後痛苦）。改用單向、有明確主從的過渡：

1. **Phase A 一次性匯入**：沿用現有 `NotesSyncService` 把 104 篇筆記灌進 DB（它本來就會做 upsert）。匯入完成後，**DB 即為真相**。
2. **匯出取代匯入**：之後改成 **DB → Markdown 匯出**（反過來），定期 commit 推上 GitHub。你「在別的裝置 git pull 就能看」「東西也都還是純文字 .md」的安全感**完全保留**，而且是自動的。
3. 真的想用 VS Code 改某篇？可以保留一個「重新匯入單檔」按鈕當逃生門，但日常寫作改走網頁。

> 這樣你得到「DB 可編輯（解鎖所有新功能）」+「Markdown 永遠是可攜備份（不被綁死）」兩全，而且沒有雙寫衝突。

### 2.4 風險與緩解

| 風險 | 緩解 |
|---|---|
| 匯入後 DB 才是真相，怕弄丟資料 | Markdown 原檔保留在 `筆記區/`（git 有歷史）；加 DB→md 自動匯出＝雙重備份 |
| 編輯衝突（未來多裝置） | 先單人；加 `UpdatedDateTime` 樂觀鎖；之後再考慮 CRDT |
| XSS（網頁可寫 HTML 後） | 沿用 Markdig `DisableHtml()`；存 raw markdown、渲染時淨化 |

## 3. 升級後的系統樣貌：五大模組 + AI

把它想成你的「**個人作業系統 / 第二大腦**」，外殼是現成的浮動面板工作區，裡面五個模組共用同一個知識圖譜與搜尋。

### 模組 ① 知識庫 Notes（既有 Article 進化）
- 網頁內**可編輯**（Markdown 編輯器 + 即時預覽），草稿/發布狀態（`PublishedFlag` 已存在）。
- **Wiki 雙向連結**：`[[X]]` 正向跳轉已有；補上**反向連結（backlinks）面板**與**知識圖譜**視覺化。
- **標籤 Tag**（補 `Tag` / `ArticleTag` 表），補強搜尋（子字串 → 全文）。
- 一鍵 AI 美化排版（沿用 `腳本/format-md.ps1` 的 Gemini 流程，搬進後端 AI 層）。

### 模組 ② 任務 Tasks / Todo（新）
- 對應你《此專案的未來》進階功能 #2、#3：
  - 多視圖：**清單 / 看板(Kanban) / 行事曆**；多排序：預計日期 / 建立日期 / 優先度。
  - **段落級 TODO**：可在某文章、甚至某段落（沿用 anchor 機制）掛一個「從這裡繼續讀/寫」的提醒。
  - 重複任務、到期提醒。

### 模組 ③ 日記 Journal（新）
- 每日一篇，可記心情；自動彙整「今天碰過的筆記/完成的任務」當日記骨架。
- 與任務、筆記互相連結（今天的日記 ↔ 今天推進的專案）。

### 模組 ④ 捕捉 Inbox（新，**摩擦力極低**是關鍵）
- 對應《此專案的未來》超進階 #3「曾經問過的問題」與 #4「發想捕捉」：
  - 內建分類 **「曾經問過的問題」**：在 ChatGPT/Claude 問完，透過 **MCP** 把問答直接寫進一篇 Page。
  - 內建分類 **「發想」**：手機/語音/打字 → 打 API → 建立 Page（原始內容 + AI 想法兩區塊）。
  - **LINE Bot** 入口：記待辦 or 記靈感都能落進系統。
- 所有捕捉先落 Inbox，再由 AI 建議分類/標籤，你一鍵歸檔。

### 模組 ⑤ 思考畫布 Canvas（**= 開問啦**，見 §4）
- Figma 式無限畫布，與 AI 分支問答；探索完的結論「**蒸餾**」成知識庫的一篇 Note。
- 定位：Notes 是「整理好的結論」，Canvas 是「正在想的過程」。兩者互通。

### 跨模組的共用能力
- **Anchor / 範圍標註**：`Comment.AnchorType/AnchorData` 已預留；開問啦的 `anchor.ts`（render-space re-anchor）可直接移植，一次滿足「範圍留言、段落 TODO、畫重點、行內連結」。
- **AI 層**：美化、摘要、自動分類、「對這篇問 AI」一鍵開畫布。
- **自己的 MCP Server**：讓任何 AI 助理能讀寫你的知識庫（開問啦已證明此模式可行）。
- **搜尋**：全文（PostgreSQL `tsvector` 或 Meilisearch）。

### 工作區外殼怎麼把這些組起來
沿用現有浮動面板工作區，新增幾種面板：**任務面板、行事曆面板、捕捉/Inbox 面板、反向連結面板、畫布面板**。命令面板 Cmd+K 升級成「全域動作入口」（搜尋 + 新增任務/筆記/捕捉 + 跳轉）。

## 4. 併入「開問啦 (KaiWen)」：為什麼、怎麼併

### 4.1 它是什麼（已驗證）
Figma 式無限畫布的 AI 分支問答工具：

- 前端：React 18 + Vite 5 + **React Flow (@xyflow/react)** + Tailwind；節點 markdown 渲染、框選提問→可點擊文字↔回答節點（InlineLink 雙向連結）、畫重點（可切換、fuzzy re-anchor）、自由連線、右側編輯 Drawer。
- 後端：**ASP.NET Core 10 + EF Core 10 + SQLite（Code-First）**；乾淨架構（`IClock`/`IUserContext`/`IAuditable`）。
- AI：`IAiProvider` 抽象 → `ClaudeCliProvider`（claude CLI、stream-json、**resume-aware**）、`OpenAiCompatibleStreamingProvider`、`FakeAiProvider`、`ImageGenerator`；SSE 串流到瀏覽器；model health check + `ai-models.json`。
- 9 張表：`Canvas / Node / Edge / InlineLink / Highlight / AiSession / AiMessage / NodeRevision / NodeImage`。
- **內建 MCP Server**（stdio，7 工具）：`list_canvases / create_canvas / list_nodes / create_node / add_floating_note / add_highlight / search_nodes`，全部呼叫 Web API → SSE 即時更新畫布。
- 狀態：9 點重構已完成、實機 + 測試（後端 31、前端 9、E2E 10）全綠。

### 4.2 驚人的契合點（這是「該併」的硬證據）

| 面向 | ZonWiki | 開問啦 | 結論 |
|---|---|---|---|
| 後端框架 | .NET 10 + EF Core 10 | .NET 10 + EF Core 10 | **完全相同** |
| DB 命名規範 | `{Table}_{Field}`、6 審計欄、反射套用 | `{Table}_{Field}`、6 審計欄、`AuditInterceptor` | **完全相同** |
| 軟刪除 | `ValidFlag` 全域查詢過濾 | `ValidFlag` 全域查詢過濾 | **完全相同** |
| API 信封 | `ApiResponse<T>{Success,Data,Error}` | `ApiResponse{Success,Data,Error}` | **完全相同** |
| MCP / 「曾問過的問題」 | 《此專案的未來》列為待做 | **已經做好了** | 開問啦正好補上 ZonWiki 缺的那塊 |

→ 兩個專案根本是「同一家族」。把開問啦當成 ZonWiki 的「Canvas + AI + MCP」模組吸收進來，**比從零寫便宜太多**。

### 4.3 三種整合策略

| 策略 | 做法 | 優點 | 缺點 |
|---|---|---|---|
| **A 橋接** | 兩個 app 各自跑，ZonWiki 透過開問啦的 MCP/API 拉畫布結論 | 最省事、低風險 | 兩 app/兩 DB(SQLite+PG)/兩前端/兩登入，長期是兩套要維護 |
| **B 完全吸收** | 把 KaiWen.Core 實體/服務移植進 ZonWiki，SQLite→Postgres，畫布 UI 嵌進 Next.js，AI 層共用 | 真正「一套系統」、單一登入/DB | 一次性工程較大 |
| **C 分階段吸收（推薦）** | 先共用 .NET solution + Postgres + 登入 + AI 設定（後端先合），畫布 UI 漸進搬進工作區；**全程用 MCP 當通用捕捉橋** | 早出價值、風險可控、最終仍是一套 | 過渡期需短暫並存 |

**推薦 C**：理由是兩者後端同源（§4.2），後端合併幾乎只是「換 EF provider + 合 migration + 接上 ZonWiki 登入」；前端 React Flow 可以用 client component 直接嵌進 Next.js。先合後端拿到「同一個 DB、同一個帳號、同一個 AI 設定」，再把畫布當作工作區裡的一種面板/分頁慢慢搬。

### 4.4 衝突清單與解法

| 衝突 | 解法 |
|---|---|
| DB：SQLite vs PostgreSQL | 統一 Postgres。EF Core Code-First，主要是換 provider + 重生 migration；KaiWen 也是啟動時 `Database.Migrate()`，習慣一致 |
| 前端：Vite SPA vs Next.js | React Flow 在 Next.js 以 `"use client"` 動態載入即可；移植 `QaNode/CanvasView/anchor.ts/highlight.ts` |
| 連接埠：5290 vs 3000/5009/5533 | 收斂到 ZonWiki 一組埠 |
| AI 供應者設定 | 把 `ai-models.json` / `IAiProvider` 併成 ZonWiki 的 AI 層；金鑰走 user-secrets/env |
| 帳號：ZonWiki 有 User，KaiWen 走 `IUserContext`（單人 system） | 以 ZonWiki 的 Google 登入為準，`CreatedUser` 帶登入者 |
| 資料模型重疊 | `InlineLink/Highlight/anchor` 機制 ↔ ZonWiki 的 `Comment.AnchorType/AnchorData`、段落 TODO 共用一套錨點實作 |

### 4.5 可直接重用的資產（不用重寫）
- `anchor.ts`（render-space 三因子 re-anchor：AnchorText + Start/End + Prefix/Suffix）→ 範圍留言、段落 TODO、畫重點一次到位。
- `IAiProvider` + 多供應者 + SSE 串流 → ZonWiki 的 AI 層直接拿來用。
- **MCP server 模式** → 變成 ZonWiki 自己的 MCP（《此專案的未來》基礎設施 #5 的需求）。
- 設計 token / 主題系統（`color-tokens-light.ts`、WCAG 對比已驗證）→ 餵進 ZonWiki 的 OKLCH 色票。

## 5. 目標資料模型（DB-first，草案）

> 全部遵循 `{Table}_{Field}`、PascalCase、單底線、6 審計欄 + `ValidFlag` 軟刪除。

**保留 / 進化**：`User`、`Category`、`Article`（加 `IsDraft` 由 `PublishedFlag` 表達；加全文索引）、`Comment`（啟用 `AnchorType/AnchorData` 做範圍留言）。

- `Tag`、`ArticleTag`（多對多標籤）

- `CaptureItem`（Inbox：`Source(line/voice/web/mcp) / RawContent / AiContent / Status(待分類/已歸檔) / TargetArticleId?`）

**從開問啦移植**：`Canvas / Node / Edge / InlineLink / Highlight / AiSession / AiMessage / NodeRevision / NodeImage`（SQLite→Postgres）。

## 6. 分階段路線圖（提案）

| 階段 | 目標 | 內容 | 產出價值 |
|---|---|---|---|
| **A. 反轉真相 + 可編輯** | DB 變真相、網頁能寫 | 一次性匯入 → Article 寫入 API（POST/PUT/DELETE）→ 網頁 Markdown 編輯器 → DB→md 匯出備份 | 從「唯讀展示」變「真的能用」 |
| **B. 任務 + 日記 + 捕捉** | 工作/生活管理上線 | Task（清單/看板/行事曆）、JournalEntry、CaptureItem(Inbox) + 段落級 TODO（移植 anchor） | 變成「個人作業系統」雛形 |
| **C. 吸收開問啦** | Canvas + AI + 自家 MCP | 後端合進 solution（SQLite→PG、共用登入/AI）；畫布 UI 嵌進工作區；自家 MCP server | 「思考過程 ↔ 整理結論」打通、AI 全面進場 |
| **D. 多入口捕捉** | 隨手記、跨裝置 | LINE Bot、行動快速捕捉、瀏覽器擴充；全文搜尋 + 知識圖譜 | 真正零摩擦的「第二大腦」 |

每階段都可獨立交付、獨立驗證。

2. **開問啦整合深度**：A 橋接 / B 完全吸收 / **C 分階段吸收（我的推薦）**，你傾向哪個？
3. **第一刀切哪**：先做「網頁可編輯」（解鎖一切）？先做「任務/Todo」？還是先把「開問啦捕捉/MCP」接上？
