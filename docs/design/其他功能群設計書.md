## 4. 英文教練（Vertex AI Live API＋後端 WebSocket 代理）

### 4.1 架構（定案：後端代理是唯一可行）

**Vertex 沒有 ephemeral token**（該機制僅 Gemini Developer API v1alpha），瀏覽器直連 Vertex Live WS 已被逐路堵死：瀏覽器 WebSocket API 不能設 Authorization header、OAuth token 走 query param 被 Vertex 401、downscoped token（CAB）只支援 Cloud Storage。**Google 官方教學本身就是 proxy 架構**——結論：**.NET 後端 WebSocket 代理是唯一路**。

```
瀏覽器（AudioWorklet 收/放 PCM16 16kHz 入/24kHz 出）
   │  WSS /ws/coach（自家後端；Cookie 認證；經 Cloudflare Tunnel）
   ▼
.NET 10 後端 CoachProxyService（ASP.NET Core WebSocket，全新基建）
   │  wss://us-west1-aiplatform.googleapis.com/ws/google.cloud.aiplatform.v1.LlmBidiService/BidiGenerateContent
   │  Authorization: Bearer <ADC access token>（VM 附掛服務帳戶，metadata server 自動取得/刷新，零金鑰檔）
   ▼
Vertex AI Live API（gemini-live-2.5-flash-native-audio，GA）
```

取捨（誠實聲明）：
- **代價**：全新 WS 基建（碼庫 WebSocket 為 0）；**Live 模型亞洲無區域**（僅美 7 區＋歐 6 區，global 不支援）——彰化 VM ↔ us-west1 估多 **120–160ms RTT**（使用者已裁示接受）；e2-small 2GB 養代理（單人 1–2 併發、純轉發 PCM 緩衝，預估可承受，列實測）；**Cloudflare Tunnel × WebSocket 支援與逾時未確認**（Phase 3 第一件事實測；備援＝另開 Tunnel hostname 或 Tailscale）。
- **紅利**（v2 方案 A 的兩個取捨消失）：①逐字稿由代理端寫入，**100% 落 DB**；②兩個 Function Call 由**後端直接處理**（收 toolCall→寫 DB→回 toolResponse→推事件給前端渲染），不繞前端。
- **.NET 實作**：官方 Google.GenAI .NET SDK 無 Live API；Google.Cloud.AIPlatform.V1 3.70.0 **沒有 LlmBidiService client（已 grep 完整 API reference 驗證）**——用 raw `ClientWebSocket` 照官方文件化 JSON 協定實作（.NET 可自由設 header），官方 Python proxy 範例當移植藍本。ADC token 用 Google.Apis.Auth `GoogleCredential`（內建快取自動刷新；本機開發用 gcloud ADC）。

**模型**：`gemini-live-2.5-flash-native-audio`（2025-12-12 發布、**GA**、**退役日 2026-12-13**）。`gemini-3.1-flash-live-preview` 只在 Developer API、Vertex 沒有無時間表——改 Vertex 暫時鎖在 2.5 世代。模型代號一律設定值（退役前換後繼模型）。

### 4.2 連線生命週期狀態機

硬限制：單一 WS 連線約 10 分鐘（GoAway 提前 60 秒）、無 compression 時音訊 session 15 分鐘、開 `context_window_compression` 後理論無限長；**session resumption：斷線後約 10 分鐘內可帶 handle 重連，快取最長保留 24 小時**（兩個約束並存——v3.1 修正：v3 曾只寫 24h，「短斷線續接」的真實窗口是 ~10 分鐘，**確切語意列 §12.16 實測**）。

```
開課：前端連自家 /ws/coach → 後端建 Vertex WS，setup 帶：
      systemInstruction（後端組裝，含到期單字注入——代理架構下天然鎖死）
    ＋ input/output audio transcription（雙向字幕；native audio 模型不支援純 TEXT 輸出，字幕靠 transcription 產生）
    ＋ context_window_compression（slidingWindow；trigger_tokens 壓低省錢，§4.9）
    ＋ session_resumption（handle 隨時更新存 CoachSession_ResumptionHandle）
上課中：收 GoAway → 後端自行帶 handle 重連 Vertex（前端無感，字幕區短暫顯示「重新連線中…」）
短斷線（來電/切 App）：~10 分鐘內前端重連 /ws/coach → 後端帶 handle 續接；
      超過 ~10 分鐘 → 走跨日續聊路（開新 session＋SummaryText＋最近 N 句逐字稿注入重建語境）
下課：前端送結束 → 後端關 Vertex WS、Status=ended、觸發課末摘要生成（VertexAdc 文字模型）
```

**孤兒連線回收（v3.1 新增——只清 DB 不清連線是資源洩漏點）**：手機直接斷網（無 close frame）時，CoachProxyService 以 **heartbeat/ping 偵測前端死亡 → 寬限 60–120 秒 → 主動關 Vertex WS、存 handle、session 標記可續接**。「孤兒連線回收」列 §12.14 實機驗證。
**殭屍會話清理（DB 層）**：讀會話清單時懶惰修正（UpdatedDateTime 超過 2 小時的 active 降轉 ended），一個 query 條件，不用排程。

### 4.3 即時字幕與「視訊感」

- 雙向 transcription 原生支援（可帶 BCP-47 language_codes 提示）；**AI 側逐字流式**（打字機效果）、**使用者側整句送達**→「🎙 聆聽中…」占位氣泡。
- **三態狀態指示**：不做 avatar（競品實測分心、嘴型不同步是負體驗），改用老師頭像圈的聆聽中／思考中／說話中三態（脈動動畫）＋麥克風音量即時回饋——hands-free 下使用者才知道系統有沒有在聽。
- **打斷（barge-in）**：自動 VAD 內建；`interrupted:true` 時清播放佇列＋**把老師字幕截到實際已播放位置、標「（被打斷）」，落 DB 的 CoachMessage 同步截斷**。
- 收音雙模式：hands-free（預設）＋push-to-talk（競品 VAD 誤判是最大痛點，必留手動退路）。
- **在課防護**：session active 時攔路由切換與關頁（確認框）；`visibilitychange` 優雅暫停、回前景重連。
- 前端音訊層搬官方 `google-gemini/live-api-web-console`（Apache-2.0）的 AudioWorklet 模組（audio-recorder／audio-streamer／audioworklet-registry／worklets），從 CRA 移植進 Next.js client component；訊息發給自家 /ws/coach，格式與 Vertex 協定同構。

### 4.4 糾錯卡（Midoo 核心體驗）

- **產生機制**：第二個 Function Call `show_correction(original, corrected, explanation_zh, better_version)`，system prompt 明示「每次三明治回饋時同步呼叫」；後端代理收 toolCall→寫 `CoachMessage_CorrectionJson`→推事件給前端。
- **呈現**：糾錯卡插在字幕串中、與語音同步——**同一張卡內逐字 diff**（錯處刪除線、修正處高亮；競品 TalkPal 因「原句修正句分離」被判死，Speak 的同氣泡對照是正面教材）；三明治三層（肯定／修正＋why／更道地說法）可摺疊。diff 一律「顏色＋刪除線/底線＋圖示」雙載體（§11）。
- **風險註**：Vertex 非同步 FC 文件完整（`FunctionResponse.scheduling = SILENT/WHEN_IDLE/INTERRUPT`），但 LiveKit 曾在舊 preview 模型實測 scheduling 欄位被拒——**GA 模型實際行為列 §12.16 實測**；備案＝循序 FC 或課末批次補產糾錯卡。

### 4.5 「加入單字本」Function Calling 流程

```

→ Vertex 觸發 toolCall: add_vocabulary(word, context_sentence)
```

### 4.6 教練 System Prompt 骨架（Midoo 式教學法落地）

由後端組裝進 `setup.systemInstruction`（代理架構下客戶端天然改不了）。骨架（實作時全文英文，此處示意）：

```text
# Role
You are "Zon Coach", a warm, patient one-on-one English tutor on a video call.
Student: native Mandarin (Taiwan) speaker, CEFR level {level}. Speak English by default.

# Teaching method (follow ALL, in priority order)
1. Sandwich feedback：糾錯一律三層——(a) 先具體肯定內容或進步點（不空泛）；
   (b) 給完整修正句＋一句 why；(c) 給一個更道地/更進階的說法（i+1）。
   做三明治回饋時「同步」呼叫 show_correction。
2. i+1 expansion：每次比學生現有程度多推一小步——引入 1 個新片語或句型，
   立刻用學生剛說的話題示範用法，邀請學生試用一次。
   Today's due review words to weave in naturally: {dueWords[]}（不可硬塞，一次最多 1-2 個）。
3. Bilingual downgrade：學生連續 2 次卡住或明顯聽不懂 → 用繁體中文簡短解釋該卡點，
   然後立刻切回英文。中文只做「解圍」，不做對話主體。
4. Two-choice guidance：學生答不上來時，不給開放式問題，改給二選一
   （"Would you say it's exciting, or stressful?"），降低開口門檻。
5. Emotional encouragement：每 3-5 輪至少一次真誠、具體的鼓勵；學生挫折時先安撫再教學。
6. Turn length：每次回應 ≤ 3 sentences，然後把球丟回給學生（問題或邀請）。
   老師說話占比 < 40%。

# Tools
- add_vocabulary(word, context_sentence)：學生要求收藏、或你判斷值得且學生同意時呼叫，
  完成後自然帶一句 "Got it, I've saved ___ for you."
- show_correction(original, corrected, explanation_zh, better_version)：每次三明治回饋
  同步呼叫，供 App 顯示糾錯卡。

# Session context（後端動態注入）
- Previous session summary: {summaryText}
- Recent transcript tail: {recentMessages}
- Student profile: {profileNotes}
```

### 5.1 資料表

`ExpenseCategory`（獨立表，勿混用筆記 Category——語意不同、避免污染筆記分類樹）：`_Name`（種子預設一組，可增改）、`_Icon?`、`_SortOrder`；**唯一 `(UserId, Name)`＋復活慣例**（「名稱式找不到就建」在捷徑重試/並發下才不會建重複，比照 `TagConfiguration.cs:24`）。

### 5.2 語音輸入雙路

- **路 1（網頁）**：桌機 Web Speech 可用；**iOS Safari 短句品質未確認**——手機瀏覽器預設打字、麥克風鈕標實驗性。不沿用 `app/page.tsx:300` 的 `isRecording` 判 source 法（送出時辨識多已結束，source 幾乎恆為 text——前端調查點名的瑕疵），改在辨識結果事件當下記。
- **路 1b（選配，音檔直傳）**：MediaRecorder → multipart → 後端**新寫 VertexGenerateContentClient 直打 Vertex 原生 `generateContent` 帶 `inlineData` 音訊＋`responseSchema`**——轉錄＋解析一步到位（棄 Groq，也不走 Cloud STT）。成本：音訊 32 tokens/秒，一句 10 秒用 flash-lite（音訊輸入 $0.30/1M tokens）**≈$0.0002/次**，比 STT+LLM 兩步便宜一個數量級。Cloud STT 備而不用（zh-TW 在 Chirp 3 是 Preview；**初代 Chirp** 社群實測中英夾雜辨識率差、**Chirp 3 是否改善未確認**；無亞洲區域；.NET SDK Google.Cloud.Speech.V2 已 GA 隨時可接）。
- **路 2（iPhone 捷徑，主力）**：Action Button → 聽寫 zh_TW（裝置端，$0）→ POST `/api/ai/expenses`（Bearer PAT）。聽寫一次只認一種語言；「300塊在7-11」夾雜品牌詞品質**未確認**（§12.17 實測）。

### 5.4 解析 JSON Schema

```jsonc
{
  "reasoning": "string（先推理再填答，避免先選分類再自圓其說）",
  "amount": 300,                        // 台灣口語數字 few-shot（300塊/三百五/1千2）
  "currency": "TWD",
  "merchant": "統一超商",                // 正規化（小七/7-11/seven → 統一超商）
```

## 6. TTS 與筆記 Podcast 模式（Cloud TTS API 上的 Gemini-TTS）

### 6.1 供應者選型與分工

**好消息：v2 選的 Gemini-TTS 已於 2025-09-30 在 Cloud TTS API 與 Vertex AI 雙軌 GA**（gemini-2.5-flash-tts／pro-tts），30 個預建聲音、style prompt、行內語氣標記（[whispering] 等）與 Developer API 版一致——v2 的聲音/語氣/Podcast 設計全數保留，只換端點與認證。

| 引擎 | 角色 | 理由 |
|---|---|---|
| **Gemini-TTS `gemini-2.5-flash-tts` via Cloud TTS API（主引擎）** | 中文為主筆記朗讀、雙主持 Podcast | **全 GCP 唯一支援 cmn-TW 的新世代 TTS（模型 GA、cmn-TW 為 Preview 語言，Phase 2 PoC 必辦）**；30 聲＋style prompt 覆蓋「男/女/語氣/個性」；multi-speaker 恰 2 講者。吃 credits |
| **Chirp 3: HD（en-US）** | 純英文內容 | **每月 1M 字元免費**（免費層獨立於 credits）、超出 $30/1M；30 聲 15男15女；控制項只有語速/停頓（無自然語言語氣）；**沒有 cmn-TW** |
| Web Speech API（前端） | 單字/短句前景即唸 | $0、非雲端服務不違約束；絕不當 Podcast 引擎（iOS 背景/鎖屏即停） |
| 保底備援（記在案） | cmn-TW 老聲音 Standard/Wavenet（腔對音質舊）或 Chirp 3 HD cmn-CN（音質好但北京腔） | Gemini-TTS cmn-TW PoC 不過關時的退路 |

- **硬限制**：Gemini-TTS 輸入 text ≤4,000 bytes（中文約 1,300 字）、單次輸出上限約 655 秒——長筆記本來就要「切段→逐段合成→ffmpeg 拼接」（§6.3 管線正好吻合）；不吃 SSML；Long Audio Synthesis API 只確認支援 Standard/Studio 聲音，不可依賴。
- 區域：Gemini-TTS 只有 global/us/eu（無亞洲）——批次合成可接受延遲。
- 認證：Cloud TTS API 用 ADC（同 VertexAdc 的 GoogleCredential 機制）；金鑰不落地。

### 6.2 聲音選擇 UI

v1＝一個下拉選單（聲音附風格標籤文字如「男・沉穩」）＋一顆「▶ 試聽」鈕（試聽句短、合成後快取）。使用者選擇存 `User_TtsSettingsJson`（比照 `User_ShortcutsJson` 慣例，最小差異 JSON）跨裝置同步。persona 前綴/二維篩選留到真的常換聲音再做。

### 6.3 後端合成管線、快取與播放狀態機

```
筆記 Markdown
 → [LLM 口語化] VertexAdc flash-lite 產出口語稿 segments[]（含章節標題標記）→ 存 TtsAudio.ScriptJson
 → [TTS] gemini-2.5-flash-tts 按句界分段（≤4,000 bytes/段，避開 655 秒輸出上限）
 → [轉檔] ffmpeg（重用 FfmpegAudioService 的 DI 慣例；spawn 遵守 UTF-8 鐵則）
 → [快取] 檔案落 App_Data/tts-cache/＋DB metadata（TtsAudio）
 → [播放] GET /api/tts/audio/{id}（enableRangeProcessing: true——iOS 拖進度條必需）
```

**播放狀態機**：①首播 `Status=processing`——逐段合成、段檔經 `GET /api/tts/audio/{id}/segments/{n}` 供檔，前端單一 `<audio>`＋佇列換源，第一段就緒即開播（`play()` 必須在使用者手勢呼叫鏈內）；進度顯示「合成中 n/N 段」。②全段完成→後端 **ffmpeg concat 成單檔**、更新 FilePath、算 `ChaptersJson`（章節標題＋時間位移）、`Status=ready`。③之後一律單檔＋HTTP Range——總時長/跨段拖曳/±15 秒/MediaSession 鎖屏 seek 全部自然正確。

**播放器控制**：**語速＝`<audio>.playbackRate`（免費、即時、不重合成）**，合成語速降為進階選項；**章節列表**（依 ChaptersJson 點擊跳段）；**續聽位置**（localStorage 記 `{ttsAudioId: 秒數}`）。

**快取與清理**：快取鍵＝SHA-256(口語稿正規化文字＋聲音＋口語化 prompt 版本＋TTS 模型代號)；索引 `(UserId, ContentHash)` 唯一。**v1 清理＝「同 NoteId＋聲音重新合成時直接失效舊列與舊檔」**（一行 replace，順帶解決筆記編輯留孤兒檔）；不預建 LRU/排程（TTS 成本天花板 <$5/月、Opus 180KB/分鐘 2GB≈185 小時，壓力幾乎不存在），靠既有**磁碟 80% email 告警**（已上線驗證）當保險。格式：Opus/Ogg 優先（iOS ≥18.4）、AAC 64kbps 保守備選，設定值。儲存：VM 磁碟檔案＋DB metadata（2GB RAM 不塞 BLOB）；prod 比照 DataProtection KeyPath 掛持久卷。**TTS 供檔端點務必自行核對 UserId 授權**（NodeImage 的「磁碟＋供檔」慣例沒有可運行參考實作，`App_Data` 不可直接靜態公開）。

### 6.4 Markdown→口語稿轉換規則（寫進口語化 prompt）

| 元素 | 唸法規則 |
|---|---|
| 表格 | 先報「這裡有一張表，欄位是…」→ 小表逐列轉敘述、大表只唸重點/總結；「唸多細」做成筆記層級設定（摘要/完整/跳過） |
| 程式碼區塊 | 不逐字唸碼：「這裡有一段約 N 行的 C# 程式碼，功能是…」；行內 code 唸字面 |

| 標題 | 明確播報（「第二節：…」）＋停頓——同時作為章節切點 |
| 清單 | 先報「以下有五點」再逐點 |
| 連結 | 只唸錨文字，不唸 URL |

口語稿保持「純文字＋Gemini 語氣標記」（不吃 SSML）。

### 6.5 MediaSession 鎖屏控制與 iOS 邊界（誠實聲明）

`navigator.mediaSession.metadata`＋`setActionHandler`（play/pause/seek）→ 鎖屏/控制中心播放器。已知風險：社群回報 iOS 鎖屏控制「時有時無」；PWA standalone 背景音訊 WebKit bug #198277＋「暫停 30 秒後鎖屏無法續播」——**Podcast 主入口先推 Safari 分頁**，standalone 待實機驗收（**未確認** iOS 26 現況）。

### 6.6 Podcast 模式 UI

筆記詳情頁「🎧 聆聽」→ 底部迷你播放器（播放/暫停/±15 秒/語速/章節/聲音）。模式：**A. 單人朗讀（預設，忠實原文）**；**B. 雙主持人對談（進階，手動觸發）**——LLM 改寫兩人對談腳本＋Gemini-TTS multi-speaker（恰 2 講者；舊 Cloud TTS Studio MultiSpeaker 已棄用下線）一次合成，NotebookLM 已驗證此體驗。B 成本較高不預設。

## 7. 筆記返回鈕重定義

### 7.1 現況

- 詳情頁返回走 `lib/noteNav.ts` 的 sessionStorage 導覽堆疊（`notes/[...slug]/page.tsx:762-788`）；堆疊起點 fallback 已是「回 `categories[0]` 的分類頁」。
- 分類清單頁（`notes/page.tsx`）**完全沒有返回鈕**——「分類頁返回→父分類」是全新 UI。
- 分類頁 URL＝`/notes?categoryId=<guid>`；「全部」＝`/notes`；`NoteCategory.parentId` 已有（`lib/api/categories.ts:14`）且整棵樹在 SWR 快取（`useNoteCategories`）——**算父分類純前端可完成，不需新 API**。

### 7.2 兩案比較與兩個必修的洞

| 方案 | 行為 | 取捨 |
|---|---|---|
| 甲、字面語意 | 筆記頁返回**永遠**回分類頁；分類頁回父分類；根＝全部 | 失去「筆記A→內文連結→筆記B→返回應回A」的穿梭；規則最簡單可預期 |
| **乙、混合・修正版（推薦）** | 堆疊有前一項→原路返回（保穿梭）；堆疊起點→回分類階層 | 保穿梭；「返回去哪」視來路而定，語意較複雜 |

兩案都必須處理的洞（第一輪評審抓到）：
1. **「掃堆疊找最近分類」是死邏輯**：`recordNoteNav` 抵頁即推入、`getNoteBackTarget` 取 idx-1（`noteNav.ts:40-62`）——走到堆疊起點時堆疊裡只剩目前頁自己。修正：**另設獨立的「最近造訪分類」sessionStorage 紀錄**（分類頁抵達時更新、不受堆疊截斷影響）。
2. **搜尋是主要進入路徑但不是堆疊起點**：稍早逛過分類A→筆記N，再從全域搜尋開筆記M（`GlobalSearch.tsx:133` 直接 router.push）→ 堆疊=[分類A, N, M]，按返回會跳到無關的舊筆記N。修正：**從非筆記情境進入（搜尋/首頁/外部連結）視為新脈絡——進入時截斷堆疊至該筆記**，使返回直接走分類階層。

### 7.3 精確規格（乙案・修正版）

**筆記詳情頁返回鈕**（依序判斷）：
1. 編輯中 → 確認未儲存變更後退出編輯（不離頁）——維持現狀。
2. 導覽堆疊有前一項（本次進入屬筆記情境內移動；非筆記情境進入時堆疊已截斷不會命中）→ `router.push(前一項)`。
3. 堆疊起點 → 回分類階層，多分類選擇規則：
   - 3a. 「最近造訪分類」紀錄中的分類 ∈ 該筆記 categories → 回它（從哪個分類脈絡來就回哪）；
   - 3b. 否則 → `categories[0]`（後端回傳順序穩定性**未確認**——實作時核對 NoteCategory 關聯排序，必要時以關聯建立時間取最早，讓行為可預期）；
   - 3c. 無分類筆記 → `/notes`（全部）。

**筆記清單/分類頁返回鈕**（全新 UI，放清單置頂列左端）：
1. `/notes?categoryId=X` → 查 SWR 分類樹：有 `parentId` → 回父分類；無 → `/notes`（全部）。
2. `/notes`（全部）→ 不顯示返回鈕（已是最終父層）。
3. `/notes?tagId=Y` → 標籤無父層 → 回 `/notes`（需求未定義，暫定，§12.21）。

**與 Header「筆記」頁籤的關係**：`Header.tsx:47-64` 的「直跳最後看的筆記」語意獨立，不改。

## 8. iOS 快速啟動（iPhone 16 Pro / iOS 26）

### 8.1 誠實邊界（先講清楚做不到什麼）

- **英文教練鎖定下不可行（三重疊加）**：持續麥克風＋字幕需前景；PWA 退背景/鎖屏即斷音訊（WebKit #198277）；捷徑無法承載長連線 WS。最快就是「解鎖後一鍵直達」。
- **聽寫記帳不是 100% 免解鎖**：鎖定下按 Action Button 可觸發、聽寫 UI 出現在鎖屏，但真正開始要 **Face ID 一瞄**——體感「一按一瞄就講話」，POST 在背景跑完。
- **真正 100% 免解鎖只有兩種**：①無 UI 的純網路請求捷徑（開「Allow Running When Locked」）；②Voice Memos 鎖屏錄音（但捷徑歷來拿不到音檔本體；iOS 26 新 action「Find Voice Recording」能否吐檔**未確認**）。

### 8.2 三功能啟動路徑設計

| 功能 | 入口分工 | 路徑 |
|---|---|---|
| **記帳** | **Action Button** | 按下（免解鎖觸發）→ 聽寫 zh_TW（Face ID 一瞄）→ 說「我剛剛花300塊在7-11買了一本書與茶葉蛋」→ **停止聆聽=暫停時（自動結束，不按側鍵）** → 背景 POST `/api/ai/expenses` → 通知「✅ **$300** 購物・統一超商」（金額最醒目，便於立即目視核對） |
| **隨手記** | **鎖定畫面自訂按鈕**（手電筒/相機鈕可換捷徑控制項，iOS 18+） | 同構捷徑，POST 目標＝既有 `POST /api/captures`（後端零改動，補掛限流即可） |
| **英文教練** | **主畫面獨立圖示** | iOS 26 任何 URL 加入主畫面預設以 standalone web app 開啟、零 manifest 門檻（WebKit 官方部落格）→ 解鎖 → 點圖示 → 直達 `/others/coach`（首次需登入一次，§4.10） |

- **離線/失敗保護**：捷徑加 If 錯誤分支——回應非 2xx 或逾時 → 聽寫文字存剪貼簿（或備忘錄）並通知「⚠️ 未送出，已暫存」。無網路（電梯/捷運）正是行動記帳高頻情境，「伺服器端保底」前提是 POST 有送達，捷徑端也要兜底。
- 備援入口：Vocal Shortcuts（iOS 18+ 輔助使用）自訂觸發詞綁記帳捷徑（鎖定可觸發；「鎖定下 Siri 語音問答全程免解鎖」來源衝突，**未確認**）。

### 8.3 捷徑設定步驟概要（記帳為例）

1. ZonWiki 個人頁 → API 權杖 → 產生 PAT（命名「iPhone記帳」；**每支捷徑獨立命名 PAT**——token_name 自動進 ActivityLog.Source 可稽核，`ApiTokenAuthenticationHandler.cs:120-129`）。
2. 捷徑 App：①「聽寫文字」（語言 zh_TW、停止聆聽=暫停時）②「取得 URL 內容」：POST `https://zonwiki.pee-yang.com/api/ai/expenses`、Header `Authorization: Bearer <PAT>`、JSON body `{ "text": 聽寫文字, "clientRequestId": 亂數UUID, "deviceNowIso": 目前日期, "timeZone": "Asia/Taipei" }` ③If 錯誤分支（§8.2）④「顯示通知」帶解析摘要。
3. 捷徑詳細資訊 → 開「允許鎖定時執行」。
4. 設定 → Action Button → 捷徑 → 選「記帳」。
- **PAT 維運 SOP**：手機遺失或懷疑外洩→個人頁撤銷該 PAT（每支捷徑獨立 PAT，影響範圍最小）；PAT 到期日機制是否已支援列 §12.18 確認。
- 已知小坑：Action Button 綁聽寫捷徑曾有「聽寫畫面出現兩次」bug（2024 回報，iOS 26 是否仍在**未確認**）。

### 8.4 PWA 地基（一次做好，三功能共用）

現況為零（無 manifest/apple-touch-icon/service worker）。補：`app/manifest.ts`（Next.js App Router 原生支援）＋apple-touch-icon＋各功能專屬啟動 URL 的 icon。iOS **不支援 manifest shortcuts**（長按圖示選單）——「一功能一圖示＋實體鍵分工」是正確架構。Service worker/Web Push 列 v2。

| Cloud TTS Standard | 4M 字元 |

### 10.3 風險與護欄（由輕到重）

- **Budget alert 只告警不斷路**（官方明文）；**付費帳戶的 credits 扣完會無縫轉扣卡**、不會停也不特別通知。Free Trial（未升級）帳戶例外：額度用完/到期→資源直接停止、不扣卡。
- **（v3.1 新增）Budget 必須勾「include credits」**：預設若監控「實付」，credits 覆蓋期間花費顯示 $0，50/90/100% 告警**全程沉默**，直到額度耗盡開始扣卡才爆。一律監控折抵前原始用量；另加一顆「scope 全帳戶、金額＝剩餘額度 80%」的**額度耗盡預警 budget**。
- **應用層斷路器（第一線，v3.1 新增）**：教練 WS 通道的三條硬上限（Origin 白名單／每人 1 併發／每日分鐘上限，§4.8）——這是唯一按分鐘燒錢的功能，護欄必須在應用層。
- per-service budget（Vertex／TTS 各一）＋壓低 aiplatform API quota 當前置限速。
- （選配、核彈級）budget→Pub/Sub→Cloud Run 呼叫 Billing API 解除專案 billing（全專案停擺；開源 Cyclenerd/poweroff-google-cloud-cap-billing）——要不要裝列 §12.9 裁決。
- 自建用量儀表板不做（GCP 原生機制＋應用層上限就夠）。

### 10.4 使用者必辦檢查清單（Phase 0）

1. `console.cloud.google.com/billing/credits`（Billing → Credits）逐筆確認：Credit name／Status／**Remaining value**／Type／**Scope**（有值＝限定特定服務，連 Vertex 都可能吃不到，要先驗）／**End date**。
2. Billing → Account management 確認帳戶是 **Free Trial 還是已升級付費**（決定額度用完是「停機」還是「扣卡」）。
3. ZonWiki 專案連到持有額度的 billing account；啟用 `aiplatform.googleapis.com`＋`texttospeech.googleapis.com`。
4. prod VM 服務帳戶授 `roles/aiplatform.user`；`gcloud compute instances describe` 確認 access scopes 含 `cloud-platform`——不足需**停機改 scope 再開機**（排維護窗）。Cloud TTS 的 IAM 權限一併用第 5 步實測驗證。
5. **用 VM 服務帳戶實打一次** Vertex `generateContent` 與 Cloud TTS `voices:list`（權限/scope 一次驗兩服務，順帶盤點 cmn-TW 現存聲音）。
6. 設 budgets：per-service（Vertex／TTS）＋「剩餘額度 80%」預警，全部**勾 include credits**；告警 50/90/100%。
7. 驗收分兩段：當日＝呼叫成功；**隔日**＝Billing report 確認 SKU 與 credit 折抵（帳單顯示有延遲）。
