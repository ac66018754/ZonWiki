一句話結論：這是一個「完成度遠超一般個人專案、但工程安全網嚴重不足」的系統。四大模組都能實際運作、資料隔離與軟刪除等價值觀級鐵則落實得扎實、文件量與誠實度也在水準之上——底子是好的。

但它同時具備一組會侵蝕你四個核心訴求的系統性缺口：CI 完全不跑測試就自動部署到 prod、前端 38K 行零測試、全站無 rate limiting（AI 端點直接燒錢）、無併發控制（多裝置同時編輯會靜默覆蓋）、四主題可讀性與錯誤色票有實測缺陷、以及部署腳本的 zone 疑似指向已搬走的舊 VM（自動部署可能已斷）。這些都屬於「平常沒事、出事很痛」的類別，優先度應高於任何新功能。

關於 2 則 Critical：兩則在對抗式驗證後都被建議降級——「CI 不跑測試」由 skeptic 判為 HIGH（屬流程/品質防線而非安全漏洞）；「無 DB 備份」查證後發現其實有一份「每日 pg_dump→GCS」的文件化 runbook，只是未碼化進版控、也無法從 repo 證實 prod 真的在跑，故降為 MEDIUM（運維驗證缺口）。誠實呈現：實質 Critical 為 0，但上述兩點仍是最該先動的 High/Medium。

1

測試與 CI 是最大系統性破口

CI（build.yml）完全不跑 dotnet test，deploy.yml 只要 build 成功就自動 SSH 進 VM 部署——5.9K 行後端測試（含 934 行資料隔離）從不在管線把關，且無 pull_request 觸發，開 PR 不跑任何檢查。更糟的是命名為「EndpointsTests」的最大宗測試根本沒測到 endpoint：不發 HTTP、只在 InMemory DB 上做 CRUD，等於在測 EF Core 本身；其中還有「設 X 讀回還是 X」的同義反覆測試。前端 38K 行則是零測試、零測試工具。README 三處宣稱的「覆蓋率 ≥ 80%」在工具鏈上完全沒有落實，是無法兌現的口號。

部署・可觀測性・成本・韌性

以「單人維護、低成本」的定位看，幾個關鍵點做得比多數個人專案好：DataProtection 金鑰環的容器持久化被正確解掉、日誌有硬性磁碟上限、Cloudflare Tunnel 不開公網埠、映像在 CI 建好推 GHCR。最致命的缺口是交付品質與資料保全：CI 無測試閘門即自動部署、部署 zone 疑似指向舊 VM、全站無 rate limit。

正式環境無「碼化且可驗證」的自動 DB 備份
CriticalPLAUSIBLEresilienceS
docker-compose.prod.yml:9-10（postgres bind-mount ./pgdata）· repo 無備份排程腳本

prod 資料只在單台 VM 的 bind-mount，repo 內無自動化 pg_dump 排程。建議：每日 pg_dump→gzip→GCS（保留 7-30 份，<NT$5/月）＋成功/失敗告警，實地演練一次還原；再加 GCE 磁碟快照作第二層。

驗證修正 → 建議降 MEDIUM：查證後發現其實有一份「每日 pg_dump→GCS + PC 離線、保留 14 份、異地三份」的文件化 runbook——並非「完全無備份」。真問題是：未碼化進版控、無法從 repo 證實 prod 真的在跑、且無告警（「備份沒跑卻沒人知道」）。應實地 crontab -l 核對最近一份 dump 日期並加告警。
CI 無測試閘門，且 build 成功即自動部署到 prod
HighCONFIRMEDcicdS
build.yml（無 dotnet test）· deploy.yml:14-26（workflow_run 成功即部署）

5.9K 行 xUnit（含 934 行資料隔離）從不在管線把關；能編譯就直上 prod。建議：build 前加 dotnet test job（Testcontainers 於 ubuntu runner 可用）作必過閘門，deploy 的 if 改為要求測試通過；前端至少加 tsc+eslint job。

部署 workflow 的 zone 疑與已遷移的 VM 不符，自動部署可能已失效
HighCONFIRMEDcicdS
deploy.yml:39-43（--zone=us-central1-a）· VM 已於 2026-07-01 遷 asia-east1-b

gcloud ssh 需正確 zone；不符會讓 Deploy job 紅燈失敗——「新映像已 build、VM 沒被重啟拉新版」。建議：更新 zone 為 asia-east1-b，把 project/zone/instance 抽成 repo variable，並補「部署後 curl /healthz 或比對版本」的健康探測步驟，失敗即紅燈。

啟動即自動 migrate 但無回滾策略，且 :latest + image prune 讓回退困難
HighPLAUSIBLEresilienceM
Program.cs:80（MigrateAsync）· deploy.yml:45（pull latest → up -d → image prune -f）

migration 失敗=全站起不來。建議：compose 改用不可變 sha tag（可 pull ...:<舊sha> 回退）；migration 與啟動解耦（CI 步驟先跑或包 try/catch + 告警）；破壞性 migration 前先自動 dump。

驗證修正 → 建議降 MEDIUM：build.yml 其實同時推不可變 sha tag（永存 GHCR），prune 只清本機 dangling、不影響 registry，可用舊 sha 回退（只是需手動改 compose）；pgdata 為持久卷，migration 失敗是停機而非資料遺失。
全站無 rate limit，AI 與認證端點無節流
HighCONFIRMEDcostM
Program.cs（無 RateLimiter）· MapAiEndpoints/MapRefineEndpoints/MapAuthPasswordEndpoints/PAT

PAT 與 AI 端點觸發付費 LLM（timeout 到 600 秒），密碼登入無嘗試次數限制。建議：先對 /api/auth/login 加匿名 IP 節流＋失敗鎖定（安全面），再對 AI/Refine 加 per-user/per-token 節流或每日額度（成本面）。（與後端 #1 同一議題。）

prod compose 對 api/web 無健康探測，restart 策略抓不到「假死」
MediumCONFIRMEDresilienceS
tmp/docker-compose.prod.yml:22-61（api/web 皆無 healthcheck）

只有 postgres 有 healthcheck；程式卡死但行程還在（執行緒耗盡/連線池枯竭/SSE 阻塞）不會被重啟。建議：外加免費 uptime 監控（UptimeRobot/GCP uptime check）打公開網域失敗發告警；純 compose 因 restart 只認退出碼，需 autoheal sidecar 或外部探針觸發重啟。

SseHub 為記憶體單例且畫布頻道永不回收，阻礙水平擴充
MediumCONFIRMEDarchitectureM
Realtime/SseHub.cs:27-42（AddSingleton；_canvases 只 GetOrAdd 從不移除，每畫布 500 筆 backlog）

SSE 狀態全在單行程記憶體、CanvasChannel 只增不減。要多實例水平擴充/藍綠部署時 SSE 無法跨實例（需 backplane），把架構鎖死在單機；記憶體慢速有界膨脹。建議：短期加閒置頻道回收（無訂閱者且逾時即移除）；要多實例時換 Redis pub/sub 或 Postgres LISTEN/NOTIFY。（訂閱者本身有正確回收。）

yt-dlp 於 build 期抓 releases/latest 無版本鎖與校驗，ffmpeg 撐大映像
MediumCONFIRMEDcostS
src/ZonWiki.Api/Dockerfile:23-27

wget latest yt-dlp 無釘版本、無 checksum＝不可重現建置＋供應鏈信任點；ffmpeg+python 撐大 API 映像。建議：釘定 yt-dlp 版本＋SHA256 校驗；把 ffmpeg/yt-dlp 這類「精煉」重依賴拆成獨立 sidecar/worker 映像，核心 API 保持精瘦。

DataProtection 金鑰環持久化但未加密於磁碟（無 ProtectKeysWith）
MediumCONFIRMEDsecurityM
Infrastructure/DependencyInjection.cs:77-79（僅 PersistKeysToFileSystem）

金鑰環以明文 XML 存在卷；它保護的正是登入 Cookie 與 DB 內 AI 金鑰密文。若 VM 被入侵/卷外洩，可解開所有金鑰、偽造 Cookie。建議：此項對「完整 VM 淪陷」加分有限（主鑰仍在同檔案系統）；真正有效的是外接 GCP KMS 保管主鑰。至少限制 dpkeys 目錄權限、確認不被納入會外流的備份/映像。

prod CORS 未設定，靠預設回退到 localhost:3000
LowCONFIRMEDresilienceS
Program.cs:38-52（回退 ["http://localhost:3000"]）· prod 未設 Cors__AllowedOrigins

目前能運作是因前後端同源（不觸發 CORS）；哪天拆不同子網域或加第三方前端，帶認證的跨域請求會突然全失敗。這是「過度限制」而非資安風險。建議：prod 環境變數明設 Cors__AllowedOrigins 為正式網域讓設定顯性化。
