# 工作流程 — 你跟 AI 怎麼一起做事

> 這份按**時間序**寫，告訴你每個情境下「你先做什麼、AI 接著做什麼」。
> 規範文件（`_rules/*`）告訴 AI **怎麼寫**，這份告訴**你跟 AI 怎麼接力**。

最後更新：2026-04-26

---

## 情境 A：全新專案 — 從零開始

### A1. 一次性準備（每個專案開頭做一次）

| 步驟 | 誰 | 動作 |
|---|---|---|
| 1 | **你** | 把 `筆記區/AI/AI-SLDC文件群-zhtw/_rules/` 整個複製到 `<專案>/docs/_rules/` |
| 2 | **你** | 在 `docs/` 建空的 `current-state.md`（用 `current-state-writing.md` 模板，Phase 設 Planning，其他 "none yet"）與空的 `constraints.md` |
| 3 | **你** | 把母本根目錄的 `AI-自治授權-範本.md` 複製到 `<專案>/docs/AI-自治授權.md`，依該專案信任 AI 的程度填入「是／否」（每個項目都要填） |

### A2. 第一次跟 AI 開工 — 產 PRD

| 步驟 | 誰 | 動作 |
|---|---|---|
| 3 | **你** | 第一句話：「按 AI-SLDC 流程開工。先讀 `docs/_rules/claude.md`、`ai-rules.md`、`current-state.md`、`constraints.md`。再開始做：**<產品想做什麼>**」 |
| 4 | **AI** | 跑完 Session Recovery Flow → 偵測無 PRD → 依 `prd-writing.md` 寫 `docs/prd.md`（Status: Draft） |
| 5 | **你** | Review PRD：欄位齊不齊、Goals 是不是 measurable、Out of Scope 寫對沒。OK 就把 `Status` 改 `Approved`，補上日期 |

### A3. 切 task

| 步驟 | 誰 | 動作 |
|---|---|---|
| 6 | **你** | 「PRD 已 approved，依 `task-writing.md` 切 task（每個對應 PRD 的 Requirement ID）」 |
| 7 | **AI** | 在 `docs/tasks/` 產出多份 task 檔（Status: Draft、Steps 寫到 concrete 等級、DoD、Verification、Related 連到 PRD F1/F2） |
| 8 | **你** | 逐份 review task：Steps 夠不夠具體、DoD 可不可量測。OK 就改 `Status: Approved`，填 `Approved By` 跟 `Approved Date` |

### A4. 執行 task（每個 task 一個循環）

| 步驟 | 誰 | 動作 |
|---|---|---|
| 9 | **你** | 「執行 task `2026-04-26-xxx.md`」 |
| 10 | **AI** | 把 task `Status` 改 `In Progress`、更新 `current-state.md` 的 `In Progress` |
| 11 | **AI** | 依 Steps 動手寫 code |
| 12 | **AI** | 跑 Verification 指令（dotnet test / pnpm lint 等） |
| 13 | **AI** | 結果寫進 task 的 Verification 段、把 `Status` 改 `Done`、`current-state.md` 把這條從 `In Progress` 移到 `Recently Completed` |
| 14 | **AI** | 回報你：「本次做完什麼／commit hash／下一步」 |
| ↻ | | 重複 9–14 直到所有 task Done |

### A5. Session 結束（**每次都做**）

| 步驟 | 誰 | 動作 |
|---|---|---|
| 15 | **你** | 「現在收工。」 |
| 16 | **AI** | 在 `docs/session-logs/YYYY-MM-DD-HHmm.md` 寫一筆 log（Worked On / Verification Run / Uncommitted Work / **Next Session Should**） |
| 17 | **AI** | 再次更新 `current-state.md`（即使這次什麼都沒完成） |
| 18 | **AI** | 列未 commit 檔案、條列下一步具體指令 |

### A6. 下次開 session

| 步驟 | 誰 | 動作 |
|---|---|---|
| 19 | **你** | 重複 §A2 第 3 步那句指令；AI 會自動讀到最新 session log，知道從哪接 |

## 情境 B：中途導入（既有 codebase 已開發）

| 步驟 | 誰 | 動作 |
|---|---|---|
| 1 | **你** | 複製 `_rules/` 到 `<專案>/docs/_rules/`，建空的 `current-state.md`（保持初始模板狀態） |
| 2 | **你** | 「按 AI-SLDC 流程開工，依 `docs/_rules/retrofit-guide.md` 回填」 |
| 3 | **AI** | 偵測到「current-state 是初始模板但有 codebase」→ 自動進入 retrofit 模式 |
| 4 | **AI** | 盤點 codebase → 回填 `prd.md`、`decisions/*`、`constraints.md`、`schema-changes/*`、`tasks/*`（已完成的標 Done、進行中的標 Draft）、`onboarding/quick-start.md`、第一筆 `session-logs/*` |
| 5 | **AI** | 在每個「原因不詳」欄位明確標註，請你補上 |
| 6 | **你** | Review 所有回填產物，補上不詳欄位、把 PRD 改 `Approved`、把已完成 task 改 `Done` 確認、把進行中 task 改 `Approved` |
| 7 | **接回 §A4** | 從步驟 9 開始正常開發循環 |

## 情境 C：修 bug

| 步驟 | 誰 | 動作 |
|---|---|---|
| 1 | **你** | 「**<bug 現象>**。按流程處理。」 |
| 2 | **AI** | 比對 `issue-writing.md` 開單門檻：達門檻 → 開 issue（Status: Open、填 Severity、Reporter、Reproducibility）；不達門檻 → 直接跟你說「這只是 X，不開 issue」 |
| 3 | **AI** | 調查 → 找 root cause → 寫進 issue 的 `Root Cause` 與 `Investigation Process` |
| 4 | **AI** | 依 `task-writing.md` 開**修復 task**（Status: Draft、Related 連到 issue） |
| 5 | **你** | Approve 修復 task（改 Status、填 Approved By/Date） |
| 6 | **AI** | 執行修復 → 跑 Verification → task Status: Done → issue Status: Resolved |
| 7 | **AI** | 若此 bug 揭示新規則 → 把規則寫進 `constraints.md`（連到 issue 為來源） |
| 8 | **AI** | 更新 `current-state.md` |

## 情境 D：架構決策（你在猶豫 A 還是 B）

| 步驟 | 誰 | 動作 |
|---|---|---|
| 1 | **你** | 「我考慮 **<選 A / 選 B>**，依流程處理」 |
| 2 | **AI** | 依 `decision-writing.md` 寫 decision（Status: Proposed、列 Options、Pros/Cons、初擬 Reason、評估 Reversibility） |
| 3 | **你** | 拍板 → 改 `Status: Accepted`、填 `Decision Maker`、`Date Decided`、確認 `Reversibility` |
| 4 | **AI** | 把 decision 的 `Rules` 同步寫進 `constraints.md`（每條附 decision 連結為來源） |
| 5 | **AI** | 若此 decision 取代舊 decision：把舊 decision 改 `Status: Superseded`、回填 `Superseded By`（雙向連結） |
| 6 | **AI** | 若 decision 影響進行中的 task → 在受影響 task 的 `Related` 加連結，必要時 task 退回 Draft |

## 情境 E：DB Schema 變更

| 步驟 | 誰 | 動作 |
|---|---|---|
| 1 | **你** | 「要加 **<表/欄位>**，依流程處理」 |
| 2 | **AI** | 依 `schema-change-writing.md` 寫紀錄（Status: Planned） |
| 3 | **AI** | 跑命名合規檢查（PascalCase 表名、`{Table}_{Field}` 欄位、六個審計欄位）— 不過就停下來告訴你 |
| 4 | **AI** | 寫 Rollback 計畫、評估 Reversibility |
| 5 | **你** | Review schema-change 紀錄，OK 才放行 |
| 6 | **AI** | 產 migration → 套到 dev → Status: Applied (Dev) |
| 7 | **AI** | 跑 migration test |
| 8 | **你** | 確認 dev 沒事 → 拍板可上 prod |
| 9 | **AI** | 套到 prod → Status: Applied (Prod)、填 `Applied On.Prod` |

## 情境 F：寫測試（新功能 / bug 迴歸）

| 步驟 | 誰 | 動作 |
|---|---|---|
| 1 | **AI** | 依 `test-writing.md` 開測試文件（Status: Planned，列 Test Cases） |
| 2 | **AI** | 寫測試 code、跑一次 → Status: RED（如預期失敗） |
| 3 | **AI** | 動實作 → 跑測試 → Status: GREEN |
| 4 | **AI** | 把驗證結果寫進對應 task 的 Verification 段與 session log |
| 5 | **AI** | 連續通過下一個 session 仍未 flake → Status: Stable |

> 注意：DB 測試一律用真實 DB（Testcontainers）。AI 若提議用 mock，立刻喊停。

## 「我該插手」的時機（依授權檔決定）

每次 AI 走到「等人類」的暫停點時，**會先重讀 `docs/AI-自治授權.md`**（依 `_rules/autonomy-authorization.md`），再決定停下還是自走。所以**這張表是「授權檔填『否』時你必須做的事」**：

| 觸發 | 對應授權項 | 授權「否」時你必須做 |
|---|---|---|
| AI 寫完 PRD Draft | #1 | 把 Status 改 `Approved` |
| AI 寫完 task Draft（含 bug fix task） | #2 | 把 Status 改 `Approved`、填 Approved By/Date |
| AI 寫完 decision Proposed（Easy） | #3 | 把 Status 改 `Accepted`、填 Decision Maker/Date Decided |
| AI 寫完 decision Proposed（Hard） | #4 | 同上 |
| AI 寫完 decision Proposed（One-Way） | #5 | 同上（**強烈建議永遠保持「否」**） |
| AI 偵測到 Material Steps Drift | #6 | 對退回 Draft 的 task 重新 Approve |
| AI 寫完 schema-change Planned 想動 dev | #7 | 放行 |
| AI 寫完 schema-change Applied (Dev) 想動 prod | #8 | 拍板放行（**強烈建議永遠保持「否」**） |
| Retrofit 回填 Draft 完成 | #9 | review 所有回填、補不詳欄位、把 Draft 提 Approved |
| AI 回報「歧義／需要你選」 | #10 | 給出選擇或補資訊 |

**授權「是」的項目** AI 會自走，並在文件對應欄位填 `AI Agent (per autonomy-authorization #N)`、在 session log 註記。修改授權檔不需要任何儀式，改完就生效，下個暫停點立刻採用新值。

**永遠不可授權**（即使授權檔填「是」也不繞過）：命名規則違規、DB mock、刪歷史文件、跳過 session log、達門檻錯誤不開 issue、decision 不同步 constraints、悄悄偏離 Steps。

## 一頁速查（給未來健忘的我）

```
你          AI
│
├─ 「按 AI-SLDC 流程開工，做 X」
│           ├─ 讀 claude.md / ai-rules.md / current-state / constraints / 最新 session log
│           ├─ 產 PRD/task/decision/issue/schema-change Draft
│           ↓
├─ Review、Approve（改 Status、填姓名日期）
│           ↓
│           ├─ 執行 → Verification → 更新 current-state.md
│           ↓
├─（給回饋或讓它繼續）
│           ↓
├─ 「現在收工」
│           ├─ 寫 session log（含 Next Session Should）
│           ├─ 更新 current-state.md
│           └─ 回報：做完什麼／未完／下一步
↓
（下次 session 從第一步重來，AI 會自動接續上次的 Next Session Should）
```
