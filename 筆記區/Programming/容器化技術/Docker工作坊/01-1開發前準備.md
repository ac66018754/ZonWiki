# 01 — 開發前準備

> 場景：主管說「把開發環境建起來，讓大家都能跑」

這個階段的目標：**任何人拿到這個專案，執行一個指令就能把整個系統跑起來。**

要達到這個目標，你需要寫幾份設定檔。這份文件會教你每份檔案是什麼、為什麼需要它、以及怎麼一行一行寫出來。

---

## 步驟一：決定專案結構

在寫任何設定之前，先想清楚專案資料夾要長什麼樣子。

**思考方式**：每個「你自己寫的程式」要有自己的資料夾，裡面放程式碼和 Dockerfile。現成軟體（postgres、redis）不需要資料夾，只要在 compose 裡寫一行就好。

```
bruce-task-manager/
├── bruce-frontend/         ← React 專案（你寫的）
│   └── Dockerfile          ← 告訴 Docker 怎麼打包前端
├── bruce-backend/          ← .NET API 專案（你寫的）
│   └── Dockerfile          ← 告訴 Docker 怎麼打包後端
├── nginx/                  ← 只放設定檔，沒有 Dockerfile
│   └── nginx.conf          ← 告訴 Nginx 怎麼分流請求
├── docker-compose.yml      ← 把所有服務串在一起
├── .env                    ← 你的真實帳號密碼（不能上 git）
└── .env.example            ← 範本，讓新人知道要填哪些變數
```

> **規則**：只有「你自己寫的程式」才需要 Dockerfile。
> 現成軟體（postgres、redis、nginx）直接在 `docker-compose.yml` 寫 `image: xxx`，
> Docker 自己去下載，不需要 Dockerfile。

---

## 步驟二：寫 Dockerfile

Dockerfile 是一份**食譜**，你告訴 Docker：「要跑我的程式，你要先做哪些事」。

每一行都是一個指令，Docker 從上到下依序執行。

### 常用指令說明

| 指令 | 意思 |
|------|------|
| `FROM` | 以哪個現成 Image 為基礎開始 |
| `WORKDIR` | 設定工作目錄（之後的指令都在這個路徑下執行） |
| `COPY` | 把本機的檔案複製進容器裡 |
| `RUN` | 在容器裡執行一個 shell 指令 |
| `ENTRYPOINT` | 容器啟動時要執行的主程式 |

---

### bruce-backend/Dockerfile（.NET API）

**思考過程**：要跑 .NET 程式，需要 .NET 環境。微軟官方提供了兩種 Image：
- `dotnet/sdk` — 完整開發工具，可以編譯程式，但很大
- `dotnet/aspnet` — 只能執行，不能編譯，但很小

所以策略是：**先用 sdk 編譯，再用 aspnet 執行**，這叫做 Multi-stage Build。

```dockerfile
# === 第一階段：編譯 ===
# FROM：以微軟官方的 .NET 8 SDK Image 為基礎
# AS build：幫這個階段取名叫 "stage1_build"，後面會用到
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS stage1_build

# WORKDIR：在容器裡建立 /app 資料夾並進入
# 之後的指令都在 /app 裡執行
WORKDIR /app

# COPY . . ：把本機目前資料夾的所有東西，複製到容器的 /app
COPY . .

# RUN：執行 .NET 的發布指令，把程式編譯成可執行檔
# 編譯結果會輸出到 /out 資料夾
RUN dotnet publish -c Release -o /out


# === 第二階段：執行 ===
# 重新 FROM，換一個更小的 Image（只能跑，不能編譯）
FROM mcr.microsoft.com/dotnet/aspnet:8.0

WORKDIR /app

# 從「第一階段（stage1_build）」的 /out，把編譯好的結果複製過來
# 這樣最終的 Image 裡沒有 SDK，只有編譯好的程式，體積小很多
COPY --from = stage1_build /out .

# ENTRYPOINT：容器啟動時執行這個指令
ENTRYPOINT ["dotnet", "BruceTaskManager.Api.dll"]
```

### bruce-frontend/Dockerfile（React）

**思考過程**：React 的運作分兩個階段：
1. **Build 時**：需要 Node.js 把 React 程式碼編譯成靜態 HTML/CSS/JS 檔案
2. **執行時**：靜態檔案只需要一個網頁伺服器來提供，不再需要 Node.js

所以一樣用 Multi-stage Build，先用 Node.js 編譯，再用 Nginx 提供服務。

```dockerfile
# === 第一階段：編譯 React ===
FROM node:20-alpine AS build

WORKDIR /app

# 先只複製 package.json，讓 Docker 可以快取這一層
# 這樣只要 package.json 沒變，下次 build 就不用重新 npm install
COPY package*.json .
RUN npm install

# 再複製其他程式碼
COPY . .

# 執行 React 的 build 指令，產出靜態檔案到 /app/dist
RUN npm run build


# === 第二階段：用 Nginx 提供靜態檔案 ===
FROM nginx:alpine

# 把 React build 出來的靜態檔案，放到 Nginx 預設的網頁目錄
COPY --from=build /app/dist /usr/share/nginx/html
```

## 步驟三：寫 nginx/nginx.conf

Nginx 是反向代理，負責當系統的「前台接待」：所有請求都進來給它，它再決定要轉給誰。

**為什麼需要這個設定檔？**
Nginx 預設只會 serve 靜態檔案，你要告訴它分流規則：什麼樣的請求要轉給後端 API，什麼樣的要轉給前端。

```nginx
server {
    # 監聽 80 port，所有進來的請求都由這個 server 區塊處理
    listen 80;

    # location：定義「什麼樣的網址」要做什麼事
    # /api/ 開頭的請求 → 轉給後端
    location /api/ {
        # proxy_pass：把請求轉發到這個地址
        # 注意：backend 是服務名稱，Docker 內部網路會自動解析成正確的 IP
        proxy_pass http://bruce-backend:8080;
    }

    # 其他所有請求（/、/tasks、/login 等）→ 轉給前端
    location / {
        proxy_pass http://bruce-frontend:80;
    }
}
```

**重點**：`http://bruce-backend:8080` 裡的 `bruce-backend` 是你在 `docker-compose.yml` 裡幫服務取的名字，Docker 會把它解析成對應容器的 IP，你不需要知道實際的 IP 是什麼。

## 步驟四：寫 docker-compose.yml

Dockerfile 只描述**單一服務**怎麼打包，docker-compose.yml 負責**把所有服務串在一起**，讓它們能互相溝通。

**常用欄位說明**

| 欄位 | 意思 |
|------|------|
| `image` | 直接從 Docker Hub 使用這個 Image |
| `build` | 用指定資料夾裡的 Dockerfile 來建立 Image |
| `ports` | 把本機的 port 對應到容器內的 port，格式是 `本機:容器` |
| `environment` | 設定容器的環境變數 |
| `volumes` | 掛載檔案或資料夾，可以是 Named Volume 或 Bind Mount |
| `depends_on` | 這個服務要等哪些服務先啟動 |

**思考方式**：每個服務寫成一個區塊，從上往下問自己：
1. 這個服務用的是現成 Image，還是我自己的 Dockerfile？
2. 它需要哪些環境變數？（帳號密碼、連線字串）
3. 它的資料需要持久化嗎？（需要的話加 Volume）
4. 它需要對外開放 port 嗎？
5. 它需要等哪些服務先起來？

```yaml
services:

  # ─────────────────────────────────────────
  # bruce-db：資料庫，用 Docker Hub 的官方 postgres Image
  # ─────────────────────────────────────────
  bruce-db:
    image: postgres:16   # 直接用官方 Image，不需要 Dockerfile

    environment:
      # postgres 這個 Image 規定用這三個環境變數來初始化資料庫
      # ${BRUCE_DB_USER} 的意思是：從 .env 檔讀取 BRUCE_DB_USER 這個變數的值
      POSTGRES_USER: ${BRUCE_DB_USER}
      POSTGRES_PASSWORD: ${BRUCE_DB_PASSWORD}
      POSTGRES_DB: bruce_taskmanager   # 容器啟動時自動建立這個資料庫，後端連線字串的 Database= 要跟這裡一致

    volumes:
      # Named Volume：資料庫的資料要持久化，否則容器刪掉資料就不見了
      # 格式：volume名稱 : 容器內的路徑
      # /var/lib/postgresql/data 是 postgres 預設存資料的地方（官方文件上會寫）
      - bruce-postgres-data:/var/lib/postgresql/data

  # ─────────────────────────────────────────
  # bruce-cache：快取，用 Docker Hub 的官方 redis Image
  # ─────────────────────────────────────────
  bruce-cache:
    image: redis:7-alpine   # alpine 是精簡版，體積更小

    volumes:
      - bruce-redis-data:/data    # redis 預設把資料存在 /data

  # ─────────────────────────────────────────
  # bruce-proxy：反向代理，用 Docker Hub 的官方 nginx Image
  # ─────────────────────────────────────────
  bruce-proxy:
    image: nginx:alpine

    ports:
      # 只有這個服務需要對外開 port
      # 本機的 80 port → 容器內的 80 port
      # 使用者打 http://localhost 就會進到這裡
      - "80:80"

    volumes:
      # Bind Mount：把本機的 nginx.conf 掛到容器內的設定路徑
      # 格式：./本機路徑 : 容器內路徑
      # 這樣改設定檔不需要重建 Image
      - ./nginx/nginx.conf:/etc/nginx/conf.d/default.conf

    depends_on:
      - bruce-frontend
      - bruce-backend

  # ─────────────────────────────────────────
  # bruce-backend：.NET API，用自己的 Dockerfile 建立
  # ─────────────────────────────────────────
  bruce-backend:
    build: ./bruce-backend   # 去 ./bruce-backend 資料夾找 Dockerfile 來 build

    # 沒有 ports：外面的人只能透過 bruce-proxy 進來，不能直接連這個服務
    # 這樣更安全，資料庫、快取也一樣不直接暴露

    environment:
      # 告訴後端程式怎麼連資料庫
      # Host=bruce-db 的 bruce-db 是上面 bruce-db 服務的名稱，Docker 會自動解析
      ConnectionStrings__Default: Host=bruce-db;Database=bruce_taskmanager;Username=${BRUCE_DB_USER};Password=${BRUCE_DB_PASSWORD}
      # 告訴後端程式怎麼連 Redis
      Redis__ConnectionString: bruce-cache:6379

    depends_on:
      - bruce-db      # 後端要等資料庫先啟動
      - bruce-cache   # 後端要等 Redis 先啟動

  # ─────────────────────────────────────────
  # bruce-frontend：React，用自己的 Dockerfile 建立
  # ─────────────────────────────────────────
  bruce-frontend:
    build: ./bruce-frontend

    depends_on:
      - bruce-backend

# 在最外層宣告用到的 Named Volume
# 不寫這個，上面的 volumes 設定會報錯
volumes:
  bruce-postgres-data:
  bruce-redis-data:
```

完整的流量走向：

```
使用者瀏覽器
    ↓  http://localhost（port 80）
  bruce-proxy（Nginx）
    ├── /api/* → bruce-backend（.NET API）
    │               ↓ 先查快取
    │             bruce-cache（Redis）
    │               ↓ 快取沒有才查
    │              bruce-db（PostgreSQL）
    └── /*     → bruce-frontend（React 靜態檔案）
```

## 步驟五：寫 .env 和 .env.example

你在 compose 裡用了 `${BRUCE_DB_USER}` 這種寫法，Docker 會去讀 `.env` 檔來取得實際的值。

**為什麼不直接把密碼寫在 compose 裡？**
compose 要 commit 進 git，如果密碼寫在裡面，任何能看到 repo 的人都知道你的資料庫密碼。

所以密碼獨立放在 `.env`，然後把 `.env` 加進 `.gitignore` 讓它不被 commit。

### .env（真實值，不能上 git）

```env
BRUCE_DB_USER=admin
BRUCE_DB_PASSWORD=supersecret123
BRUCE_REDIS_PASSWORD=
```

### .env.example（範本，要 commit 進 git）

這份檔案複製自 `.env`，但把所有值清空。
用途是讓新人知道「你需要設定哪些變數」，他複製後自己填入實際值。

```env
BRUCE_DB_USER=
BRUCE_DB_PASSWORD=
BRUCE_REDIS_PASSWORD=
```

### .gitignore 加這行

```
.env
```

| 檔案 | 有真實密碼 | commit 進 git | 用途 |
|------|-----------|--------------|------|
| `.env` | 是 | **不行** | Docker 執行時讀取 |
| `.env.example` | 否（值是空的） | 是 | 新人加入時的填寫範本 |

---

## 新人加入時要做什麼？

所有設定都準備好並 commit 進 git 之後，新人只需要：

```bash
# 1. 把專案抓下來
git clone https://github.com/yourcompany/bruce-task-manager

cd bruce-task-manager

# 2. 複製環境變數範本，然後打開 .env 填入實際的帳號密碼
cp .env.example .env

# 3. 啟動所有服務
docker compose up
```

Docker 在 `compose up` 的時候會自動：
- 從 Docker Hub 下載 postgres、redis、nginx 的 Image
- 用 Dockerfile 建立 frontend 和 backend 的 Image
- 把所有容器啟動並串接起來

打開瀏覽器 `http://localhost` 就能看到整個系統，只有一個入口。
