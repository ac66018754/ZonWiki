# Docker 工作坊 — 專案介紹

## 我們要做什麼？

假設你加入了一間公司，主管說：「我們要做一個**任務管理系統**，請你把環境建起來。」

這個系統由五個部分組成：

| 元件 | 技術 | 用途 |
|------|------|------|
| 前端 | React | 使用者操作介面 |
| 後端 | .NET Web API | 處理業務邏輯、提供 API |
| 資料庫 | PostgreSQL | 長期儲存資料 |
| 快取 | Redis | 暫存常用資料，減少資料庫查詢次數 |
| 反向代理 | Nginx | 統一入口，把請求分流到前端或後端 |

---

## 沒有 Docker 以前，會發生什麼事？

1. 新人加入 → 要裝 Node.js、.NET SDK、PostgreSQL
2. 各自版本不同 → A 的電腦跑得起來，B 的電腦壞了
3. 測試環境跟正式環境設定不一樣 → 測試沒問題，上線爆炸
4. 要換一台機器部署 → 重新裝一遍所有環境

這就是「**在我電腦上是好的**」的問題。

---

## Docker 怎麼解決這件事？

Docker 把「程式」和「執行環境」打包在一起，變成一個 **Image**。

- 不管在誰的電腦、哪台伺服器，跑起來的行為都一樣
- 新人加入只需要裝 Docker，`docker compose up` 就全部啟動

---

## 這個專案需要哪些 Image？

```
任務管理系統
├── frontend        ← 自己寫 Dockerfile 建立（React）
├── backend         ← 自己寫 Dockerfile 建立（.NET API）
├── db              ← 直接從 Docker Hub 拉官方 Image（postgres）
├── cache           ← 直接從 Docker Hub 拉官方 Image（redis）
└── proxy           ← 直接從 Docker Hub 拉官方 Image（nginx）
```

### 哪些要自己寫？哪些用現成的？

| Image | 來源 | 原因 |
|-------|------|------|
| `frontend` | 自己寫 Dockerfile | 你的 React 程式碼是獨特的，要自己打包 |
| `backend` | 自己寫 Dockerfile | 你的 .NET API 程式碼是獨特的，要自己打包 |
| `postgres` | Docker Hub 官方 | 現成的資料庫軟體，不需要客製 |
| `redis` | Docker Hub 官方 | 現成的快取軟體，直接用就好 |
| `nginx` | Docker Hub 官方 | 反向代理只需要掛設定檔，不需要自建 Image |

### 為什麼要有 Redis 和 Nginx？

**Redis（快取）**：
使用者每次打開任務列表，API 都要去查資料庫。如果同時 100 個人開，資料庫就要查 100 次一樣的東西。
Redis 把查詢結果暫存在記憶體，第二次以後直接從 Redis 拿，不碰資料庫，速度快 10 倍以上。

**Nginx（反向代理）**：
沒有 Nginx，前端和後端要開兩個 port（`:3000` 和 `:5000`），使用者要記兩個網址，瀏覽器還會有跨域問題。
有 Nginx，外面只看到一個 port（`:80`），Nginx 內部自動判斷：
- 網址是 `/api/...` → 轉給後端
- 其他 → 轉給前端
