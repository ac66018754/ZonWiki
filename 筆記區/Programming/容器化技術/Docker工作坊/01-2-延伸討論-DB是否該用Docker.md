# 01-1 — 延伸討論：DB 是否該用 Docker？

> 上一篇我們用 Docker 把整個系統跑起來，包含 PostgreSQL。
> 但實務上，DB 真的適合放在 Docker 裡嗎？

---

## 先講結論

| 環境 | DB 用 Docker？ | 原因 |
|------|---------------|------|
| **Local（開發者本機）** | 可以，甚至推薦 | 方便、一鍵啟動、不怕搞壞 |
| **QA（測試環境）** | 看情況 | 小團隊可以用，大團隊通常用獨立 DB |
| **Prod（正式環境）** | 幾乎不會 | 資料安全、效能、高可用性都不適合 |

---

## 為什麼 Prod 不用 Docker 跑 DB？

Docker 的設計哲學是：**容器是可拋棄的**。壞了就砍掉重建。

但資料庫正好相反 — 資料是最不能丟的東西。

### 1. 資料安全性

Docker 容器被刪除、重建、升級出錯，裡面的資料都可能消失。
雖然可以用 Volume 把資料掛出來，但多了一層風險。

雲端託管的 DB（AWS RDS、Azure SQL、GCP Cloud SQL）：
- 自動備份（每天甚至每小時）
- 自動異地備援（機房掛了也不怕）
- 硬碟壞了，雲端商自動處理

### 2. 效能

Docker 多了一層儲存抽象，對 DB 這種 **I/O 密集** 的服務影響很大：
- DB 需要直接、快速地讀寫硬碟
- Docker 的儲存層（storage driver）會產生額外開銷
- 正式環境的 DB 通常需要精確的記憶體和 CPU 調校

### 3. 高可用性

正式的 DB 需要一整套機制：

```
主 DB
 ├── 備援 DB（主 DB 掛了，自動切換過來）
 ├── 唯讀副本（分散查詢壓力）
 └── 自動備份（定時快照）
```

雲端託管服務內建就有這些功能。用 Docker 自己搞，要花非常大的力氣，而且容易出錯。

### 4. 維運成本

| 事情 | 雲端託管 DB | Docker DB |
|------|------------|-----------|
| 升級版本 | 點幾下按鈕 | 自己處理資料搬移 |
| 硬碟滿了 | 自動擴充 | 自己搬 Volume |
| 效能監控 | 內建儀表板 | 自己裝監控 |
| 資安修補 | 自動套用 | 自己追更新 |

---

## 那 QA 環境呢？

QA 介於兩者之間，看團隊規模和需求：

**小團隊 / 新專案**：Docker 跑 DB 是合理的
- 省錢（不用多開一台 DB 伺服器）
- 測試環境本來就可以接受資料不見
- 方便重建乾淨的測試環境

**大團隊 / 成熟專案**：通常用獨立的 DB
- 多人共用同一份測試資料
- QA 人員需要穩定的環境，不能隨便壞掉
- 可能需要跟 Prod 相近的架構來做驗證

---

## Local 開發呢？

這是 Docker 跑 DB **最合適的場景**：

- 一個指令就有 DB 可以用，不用安裝任何東西
- 想砍掉重建就砍掉重建，不影響任何人
- 可以同時跑不同版本的 DB（PostgreSQL 16 和 MySQL 8 並存）
- 不需要連外部網路也能開發

不過有些公司的做法是 **Local 直接連 QA 的 DB**，這樣就不需要本機的 DB：

```
你的電腦（跑程式）→ 連線到 → QA 的 DB 伺服器
```

好處是大家共用同一份資料，壞處是你改壞資料會影響其他人。

---

## 延伸：哪些東西不適合用 Docker？

Docker 不適合的共通點：**需要長期保存狀態、需要直接操作硬體、或需要極致效能的東西。**

| 不適合 Docker 的 | 原因 | 正式環境通常怎麼做 |
|-----------------|------|-------------------|
| **資料庫** | 資料不能丟、需要高 I/O 效能 | 雲端託管（RDS、Azure SQL） |
| **檔案儲存** | 大量檔案讀寫、需要持久化 | 雲端儲存（S3、Azure Blob） |
| **訊息佇列** | 訊息不能丟、需要叢集 | 雲端託管（Amazon MQ、CloudAMQP） |
| **搜尋引擎（Elasticsearch）** | 索引資料量大、I/O 密集 | 雲端託管（Elastic Cloud） |
| **需要 GPU 的服務** | 需要直接存取顯卡硬體 | 實體機或專用 GPU 雲端主機 |
| **桌面應用程式** | 需要圖形介面 | 不適用 Docker |

> **共通點**：只要是「重度依賴磁碟 I/O」或「資料絕對不能丟」的服務，正式環境就不適合 Docker。

---

## 那什麼最適合用 Docker？

Docker 最擅長的是：**無狀態的、可以隨時砍掉重建的服務。**

| 很適合 Docker 的 | 原因 |
|-----------------|------|
| **後端 API** | 無狀態，壞了重啟一個就好 |
| **前端（靜態網站）** | 只是提供檔案，不存資料 |
| **反向代理（Nginx）** | 只做轉發，不存資料 |
| **排程任務（Cron Job）** | 跑完就結束，不需要持久化 |
| **CI/CD 的建置環境** | 每次建置完就丟掉，天生適合容器 |
| **微服務** | 每個服務獨立部署、獨立擴充 |
| **開發和測試環境的一切** | 方便建立、方便銷毀、不怕搞壞 |

> **共通點**：程式本身不存資料（資料存在外部的 DB 或儲存服務），容器砍掉重建不會有任何損失。

---

## 一句話總結

> **Docker 擅長跑程式，不擅長存資料。**
> 正式環境把「跑程式」和「存資料」分開：程式放 Docker，資料交給專門的託管服務。
> 開發環境則無所謂，全部丟 Docker 最方便。
