# 02 — 開發中

> 場景：環境建好了，你開始每天寫程式

---

## 每天的工作流程

### 早上開始工作

```bash
docker compose up
```

所有服務（前端、後端、資料庫）一次全部啟動。  
你打開瀏覽器 `http://localhost:3000` 就可以開始工作。

### 下班離開

```bash
docker compose down
```

所有容器停止，但**資料庫的資料還在**（因為我們用了 Volume）。

---

## 我改了程式碼，容器會自動更新嗎？

**不一定**，要看你怎麼設定。

### 情況 A：有設定 Bind Mount（開發推薦）

在 `docker-compose.yml` 加上 volumes 把本機程式碼掛進去：

```yaml
bruce-backend:
  build: ./bruce-backend
  volumes:
    - ./bruce-backend:/app    # 本機的 ./bruce-backend 資料夾 = 容器內的 /app
```

這樣你在本機改了程式碼，容器內也會立刻看到改變。  
搭配 .NET 的 `dotnet watch` 或 React 的 Hot Reload，**存檔就自動重新整理**，不需要重建 Image。

### 情況 B：沒有 Bind Mount（改了要重建）

```bash
docker compose build bruce-backend   # 重新建立 bruce-backend 的 Image
docker compose up bruce-backend      # 重啟
```

---

## 只有特定服務掛掉，怎麼只重啟它？

```bash
docker compose restart bruce-backend
```

不需要把全部服務都停掉重啟。

---

## 想看 Log 怎麼辦？

```bash
docker compose logs bruce-backend         # 看 bruce-backend 的 log
docker compose logs -f bruce-backend      # 持續追蹤（像 tail -f）
docker compose logs                 # 看所有服務的 log
```

---

## 想進到容器裡面執行指令？

```bash
docker compose exec bruce-backend bash
```

就像 SSH 進去一樣，可以在容器內執行任何指令（查看檔案、手動跑 migration 等）。

```bash
docker compose exec bruce-db psql -U admin -d bruce_taskmanager   # 直接進 PostgreSQL
```

---

## 跑資料庫 Migration

通常在開發中會這樣做：

```bash
docker compose exec bruce-backend dotnet ef database update
```

在容器內執行 EF Core migration，資料庫會自動建立/更新 schema。

---

## 常見問題

### 改了 Dockerfile 或 compose 設定，要怎麼更新？

```bash
docker compose up --build
```

加上 `--build` 強制重新建立 Image，而不是用快取的舊版本。

### 資料庫壞掉想整個重置？

```bash
docker compose down -v    # -v 會同時刪掉 Volume，資料全清
docker compose up
```

> 這個動作會**永久刪除所有資料**，只在開發環境用！

---

## 小結：開發中 Docker 帶來了什麼？

| 問題 | 沒有 Docker | 有 Docker |
|------|------------|-----------|
| 新人要設定環境 | 花幾小時 | 一行指令 |
| 版本衝突 | 常發生 | 容器隔離，不互相影響 |
| 「我電腦沒問題」 | 常聽到 | 大家跑一樣的環境 |
| 切換不同專案 | 版本衝突 | 各專案各自的容器 |
