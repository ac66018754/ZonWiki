# 03 — 開發後部署

> 場景：功能做完了，要把它部署到正式伺服器上

---

## 開發環境 vs 正式環境的差異

| | 開發環境 | 正式環境 |
|---|---------|---------|
| 目的 | 方便修改、除錯 | 穩定、安全、效能 |
| Bind Mount | 有（程式碼掛進去） | 沒有（Image 裡就包含程式碼） |
| 熱重載 | 有 | 沒有 |
| Debug 模式 | 開啟 | 關閉 |
| Image | 不一定最佳化 | 要盡可能小、安全 |

---

## 步驟一：建立正式環境的 Image

```bash
# 在本機或 CI/CD 上執行
docker build -t bruce/task-backend:1.0.0 ./bruce-backend
docker build -t bruce/task-frontend:1.0.0 ./bruce-frontend
```

- `bruce/task-backend` 是 Image 名稱
- `1.0.0` 是版本標籤（tag），每次部署都要打新版本號

---

## 步驟二：推送 Image 到 Registry

**Registry** 是存放 Image 的倉庫，就像 GitHub 是存放程式碼的倉庫。

```bash
docker push bruce/task-backend:1.0.0
docker push bruce/task-frontend:1.0.0
```

常見的 Registry：
- **Docker Hub**：公開的，免費（公開 Image）
- **GitHub Container Registry (ghcr.io)**：跟 GitHub 整合
- **AWS ECR / Azure ACR / GCP Artifact Registry**：各家雲端自己的

> 這就是為什麼公司會有私有 Registry，存放不能公開的 Image。

---

## 步驟三：在伺服器上部署

伺服器上有一份 `docker-compose.prod.yml`，跟開發的 compose 不同：

```yaml
services:

  bruce-db:
    image: postgres:16
    environment:
      POSTGRES_USER: ${BRUCE_DB_USER}
      POSTGRES_PASSWORD: ${BRUCE_DB_PASSWORD}
      POSTGRES_DB: bruce_taskmanager
    volumes:
      - bruce-postgres-data:/var/lib/postgresql/data
    restart: always           # 伺服器重開機自動啟動

  bruce-backend:
    image: bruce/task-backend:1.0.0    # 用已經建好的 Image，不是 build
    environment:
      ASPNETCORE_ENVIRONMENT: Production
      ConnectionStrings__Default: Host=bruce-db;...
    restart: always

  bruce-frontend:
    image: bruce/task-frontend:1.0.0
    ports:
      - "80:80"
      - "443:443"
    restart: always

volumes:
  bruce-postgres-data:
```

在伺服器上執行：

```bash
docker compose -f docker-compose.prod.yml pull    # 從 Registry 拉最新 Image
docker compose -f docker-compose.prod.yml up -d   # -d 背景執行
```

---

## CI/CD 自動化部署

實務上，這些步驟不會手動執行，而是由 CI/CD 系統自動完成：

```
開發者 git push
    ↓
GitHub Actions / GitLab CI 觸發
    ↓
1. 跑測試
2. docker build（建 Image）
3. docker push（推到 Registry）
4. SSH 進伺服器，docker compose pull + up
    ↓
新版本上線
```

你寫的 Dockerfile 和 docker-compose.prod.yml 就是讓這個流程能自動化的關鍵。

---

## 為什麼正式環境不用 docker-compose build？

開發環境用 `build: ./backend`，讓 Docker 現場建 Image。  
正式環境用 `image: bruce/task-backend:1.0.0`，直接用已建好的 Image。

原因：
- 確保測試和部署的是**同一個 Image**（測試通過的那個）
- 伺服器不需要有原始碼，安全性更高
- 部署速度更快（Image 已經建好了）
