# 04 — 維運

> 場景：系統上線後，你或 DevOps 同事要確保它持續正常運作

---

## 系統上線後 Docker 還有什麼用？

上線只是開始，維運才是長期工作。Docker 在這個階段的角色：

1. 確保服務掛掉能自動重啟
2. 更新版本時不停機（或最小停機）
3. 查看系統狀態與 Log
4. 擴充服務應對流量

---

## 查看系統狀態

```bash
docker compose ps                # 所有服務的狀態
docker stats                     # 即時 CPU、記憶體使用量（像 top 指令）
docker compose logs -f bruce-backend   # 即時追蹤 bruce-backend log
```

---

## 更新到新版本

當有新版本的 Image 推上 Registry：

```bash
docker compose -f docker-compose.prod.yml pull       # 拉新版 Image
docker compose -f docker-compose.prod.yml up -d      # 重新啟動（有更新的服務才會重啟）
```

Docker Compose 夠聰明，**只會重啟有變動的服務**，沒改到的服務繼續跑，減少停機時間。

---

## 備份資料庫

```bash
# 把 PostgreSQL 資料備份到本機
docker compose exec bruce-db pg_dump -U admin bruce_taskmanager > backup_20240418.sql
```

還原：

```bash
cat backup_20240418.sql | docker compose exec -T bruce-db psql -U admin -d bruce_taskmanager
```

## 服務掛了怎麼辦？

### 查看是哪個服務掛了

```bash
docker compose ps
```

STATUS 如果不是 `running`，代表有問題。

### 看它為什麼掛

```bash
docker compose logs bruce-backend --tail=100   # 看最後 100 行 log
```

### 重啟

```bash
docker compose restart bruce-backend
```

### 真的嚴重，整個重建

```bash
docker compose down
docker compose -f docker-compose.prod.yml up -d
```

---

## `restart: always` 的作用

在 compose 設定裡加了這個：

```yaml
bruce-backend:
  restart: always
```

代表：
- 容器崩潰 → Docker 自動重啟
- 伺服器重開機 → Docker 重新啟動所有服務

這樣就不用人工半夜爬起來重啟服務。

---

## 進階：Kubernetes（K8s）

當你的系統規模更大，光靠 Docker Compose 不夠用時，就會用 **Kubernetes**：

| | Docker Compose | Kubernetes |
|---|---|---|
| 適合場景 | 單台伺服器、小型專案 | 多台伺服器、大規模系統 |
| 自動擴充 | 不支援 | 支援（自動增減容器數量） |
| 學習曲線 | 低 | 高 |

> 但 Kubernetes 仍然是跑 Docker Container，只是管理方式更複雜。先把 Docker Compose 搞熟，K8s 只是下一步。
